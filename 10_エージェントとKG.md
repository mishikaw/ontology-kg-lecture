## 本章のゴール

- QAボット(9章)と**エージェント**の違いを説明できる
- KGを**ツール**としてエージェントに渡す設計ができる
  - スキーマ取得+クエリ実行の2点セット、MCPによる接続
- **ガードレール**(権限・コスト・ステップ上限)を設計できる
- クエリ失敗時の**自己修正ループ**と、KGへの**書き戻し**を実装できる

> 到達イメージ:
> **「この障害の影響を受ける顧客と担当営業は?」を、エージェントが自律的にグラフを探索して解く**

<!--
- 到達イメージの質問は1章のデモで見せた質問と同じ。「あのデモの最終形を今日自分で作る」と伝えると円環が閉じる
- 「自律的に」の意味を先に定義:人間がクエリを書かない、LLMが結果を見て次の一手を決める
- ガードレールと信頼性が本章の裏テーマ。「動くエージェント」と「本番に置けるエージェント」の差はここで決まると予告
- 受講者への問いかけ:「エージェントに社内DBを触らせるとしたら、何が怖い?」→ 権限・暴走・コスト、が後半の伏線になる

## QAボットからエージェントへ

| | QAボット(9章) | エージェント(本章) |
|---|---|---|
| タスク | 単発の質問応答 | **複数ステップの調査・判断** |
| 処理の流れ | 検索→生成の**固定パイプライン** | 計画→実行→観察の**ループ** |
| クエリの決め方 | 人間が設計した経路(1回) | **LLMが結果を見て次を決める** |
| 失敗したら | そのまま低品質な回答 | リトライ・別アプローチを試す |
| 得意な質問 | 「X社の担当者は?」 | 「この障害の影響と対応優先度は?」 |

- 固定パイプラインで解けるなら9章の構成で十分(安く・速い)
- **途中結果次第で調べ方が変わる**タスクがエージェントの領分

<!--
- 最重要の対比スライド。「エージェント=偉い」ではなく「タスクの性質で使い分ける」と最初に釘を刺す
- 9章のハイブリッドGraphRAGは検索経路を人間が設計済み。設計時に想定した質問には強いが、想定外の質問には弱い
- 「影響と対応優先度」の例:影響範囲を調べた結果を見て初めて「どの顧客が大口か」を追加で調べる必要がわかる — 事前に経路を固定できない
- 最下行が11章のコスト設計にも効く:エージェントはLLM呼び出しが数倍〜十数倍になる
-->

## エージェントの基本ループ

```
        ┌──────────────────────────────────────┐
        ↓                                      │
質問 → 計画(次に何を調べるか) → ツール実行 → 結果を観察
        │                                      
        └→ 材料が揃った → 回答を合成(根拠パス付き)
```

- LLMが**ツール(関数)を呼び出す**:tool use / function calling
- ループの各周回で「続けるか、答えるか」をLLM自身が判断
- KG側に必要なのは:**エージェントが使える形のツール定義**

→ 次スライド:KGをツール化する設計

<!--
- 受講者は生成AIプログラミング経験者なので tool use 自体は既知の想定。軽く確認だけする(「function calling使ったことある方?」)
- ポイントは「ループ」であること。1回のtool useで終わるのはQAボットと同じ
- 「観察」が肝:クエリ結果(JSON)が次のプロンプトに入り、LLMの次の判断材料になる
- 「KG側に必要なのはツール定義」で視点をKG設計者側に切り替える。本章の主役はエージェントフレームワークではなくKGのツール化設計
-->

## KGをツール化する:2点セット

| ツール | 役割 | なぜ必要か |
|---|---|---|
| **① スキーマ取得** `get_graph_schema` | ノードラベル・リレーション型・プロパティ一覧を返す | LLMは**スキーマを知らないと正しいクエリを書けない**(5章) |
| **② クエリ実行** `run_cypher` | 読み取りCypherを実行し結果をJSONで返す | 探索の実行部。**読み取り専用**にする |

- この2つだけで、エージェントは自律探索できる
- スキーマをシステムプロンプトに固定埋め込みしない理由:
  - スキーマは進化する(8章)/ 大きいKGでは全量埋め込みが高コスト
  - **必要なときに取りに行く**方がトークン効率と鮮度で有利

<!--
- 「たった2つ」であることを強調。ツールを増やしすぎるとLLMの選択ミスが増える。最小から始める
- ①は5章の伏線回収:「Text2Cypherへの入力はスキーマの質で決まる」— エージェントでも同じで、スキーマ取得を最初のステップに誘導する
- ツールのdescriptionに「クエリ作成前に必ずスキーマを確認すること」と書く運用テクニックを紹介(次スライドの実物で見せる)
- 「よく使う定型探索(例:影響範囲取得)を第3のツールとして足す」発展形にも触れる:精度が要る探索はパラメータ化した関数にしてLLMの自由度を下げる
-->

## ツール定義のJSON例

```json
[
  {
    "name": "get_graph_schema",
    "description": "KGのスキーマ(ノードラベル・リレーション型・プロパティ)を返す。Cypherを書く前に必ず呼ぶこと。",
    "input_schema": { "type": "object", "properties": {} }
  },
  {
    "name": "run_cypher",
    "description": "読み取り専用のCypherを実行し、結果をJSONで返す。MATCH〜RETURN形式のみ。結果は最大50行。",
    "input_schema": {
      "type": "object",
      "properties": {
        "query": { "type": "string", "description": "実行するCypher。LIMITを必ず付けること。" }
      },
      "required": ["query"]
    }
  }
]
```

<!--
- descriptionが実質「エージェントへのマニュアル」。ここに使い方・制約・順序を書き込むのが tool use 設計の要諦
- 「必ずスキーマを先に呼ぶ」「LIMITを付ける」— 制約を自然言語で書いておくとLLMはかなり従う。ただし保証はないので後半のガードレール(サーバ側強制)と二段構えにする
- 「結果は最大50行」はコンテキスト溢れ対策。巨大な結果をそのまま返すとトークンを食い潰す
- 受講者への問いかけ:「このdescriptionに何を足したい?」→ 例:プロパティの値例、日付フォーマット、など実務の工夫が出るとよい
-->

## 一発Text2Cypher vs 逐次探索

```
■ Text2Cypher(9章)— 一発勝負
質問 ──→ [スキーマ + 質問] → LLM → Cypher 1本 → 実行 → 回答
          クエリ1本に収まらない質問・スキーマ推測ミスで失敗

■ 逐次探索(エージェント)— 結果を見て次を決める
質問 ──→ スキーマ確認
      ──→ クエリ①「障害F12は存在する?」→ 結果を観察
      ──→ クエリ②「F12が影響するシステムは?」→ 結果を観察
      ──→ クエリ③「そのシステムを使う顧客と担当営業は?」
      ──→ 材料が揃った → 回答合成(根拠パス付き)
```

<!--
- 本章の中核となる対比図。板書的にゆっくり
- Text2Cypherの失敗モードを具体的に:①複雑質問を1本のクエリに詰め込んで構文エラー ②エンティティ名の表記ゆれ(「障害F12」vs「INC-F12」)で空振り ③スキーマの誤解
- 逐次探索は各ステップが単純なクエリになる=1本あたりの成功率が上がる。さらに①で存在確認してから進むので表記ゆれにも対処できる
- 代償はLLM呼び出し回数とレイテンシ。「単純な質問には9章のText2Cypher、調査タスクにはエージェント」というルーティングが実務解(11章)
-->

---

## 逐次探索が効く場面

| 場面 | 一発Text2Cypher | 逐次探索エージェント |
|---|---|---|
| **表記ゆれ** 「F12障害」→ 実データは `INC-F12` | 空振りで終了 | まず候補検索→正式IDを特定して続行 |
| **空結果** 条件が厳しすぎた | 「見つかりません」 | 条件を緩めて再クエリ |
| **複合質問** 影響範囲+担当+優先度 | 巨大クエリで失敗しがち | 小さいクエリに分解して積み上げ |
| **途中の判断** 結果次第で調べ先が変わる | 表現不能 | 観察→計画で自然に対応 |

> 逐次探索の本質:**クエリの成功率を「1本の完璧さ」から「修正可能なループ」に置き換える**

<!--
- 前スライドの図を場面別に具体化。受講者の「Text2Cypherで頑張ればよくない?」という疑問に正面から答える
- 表記ゆれ対応は実務で最頻出。エージェントは「まず `CONTAINS` や全文検索で候補を出す」という人間と同じ手順を踏める
- 空結果からの条件緩和は諸刃の剣:緩めすぎると誤答になる。回答に「条件を緩めて検索した」旨を含めさせるのが実務の工夫
- 引用枠のテーゼは試験に出るレベルで重要。「完璧な1本」を目指すより「間違えても直せるループ」の方が総合精度が高い、はLLMシステム設計全般に通じる
-->

## MCP:ツール接続の標準プロトコル

**MCP(Model Context Protocol)** = LLMアプリと外部ツール・データ源を繋ぐオープン標準

```
LLMアプリ(ホスト)                     外部リソース
 Claude / IDE / 自作エージェント
   │
   ├─ MCPクライアント ⇄ MCPサーバA(Neo4j:スキーマ・クエリ)
   ├─ MCPクライアント ⇄ MCPサーバB(社内Wiki)
   └─ MCPクライアント ⇄ MCPサーバC(チケット管理)
```

- サーバが**ツール群を宣言**し、ホストのLLMがそれを呼び出す
- 利点:**N個のアプリ × M個のツールの個別実装が不要**に
  - Neo4j用MCPサーバは公式提供 → 自前のツール実装なしで接続できる

<!--
- MCPを初めて聞く受講者もいる想定で、「USB-Cのような共通规格」という例えが通じやすい
- さっき自分で書いたツール定義JSONとの関係を明確に:MCPサーバは「あのツール定義+実装」をパッケージ化して配布可能にしたもの。概念は同じ
- N×M問題:アプリごと・ツールごとに接続コードを書く組み合わせ爆発が、プロトコル標準化で解消される
- 「自作エージェントでもMCPクライアントライブラリを使えば公式Neo4jサーバのツールをそのまま使える」— 車輪の再発明をしない、が7章から一貫するコースの姿勢
-->

---

## Neo4j MCPサーバの設定例

`mcp-neo4j-cypher`(公式)を Claude Desktop / 自作ホストに登録:

```json
{
  "mcpServers": {
    "neo4j": {
      "command": "uvx",
      "args": ["mcp-neo4j-cypher"],
      "env": {
        "NEO4J_URI": "bolt://localhost:7687",
        "NEO4J_USERNAME": "agent_reader",
        "NEO4J_PASSWORD": "********"
      }
    }
  }
}
```

提供されるツール(まさに「2点セット+α」):
`get_neo4j_schema` / `read_neo4j_cypher` / `write_neo4j_cypher`

<!--
- ライブで設定→接続→ツール一覧が見えるところまでデモすると効果的
- 提供ツールが本章前半で設計した「スキーマ取得+クエリ実行」の2点セットと一致していることを確認。「設計原理がわかっていれば既製サーバの良し悪しも判断できる」
- write_neo4j_cypher が最初から入っていることに注意喚起 → 次スライドのガードレールへの導入。「このまま本番KGに繋いだら何が起きる?」と問いかける
- ユーザー名を agent_reader にしているのは意図的(次スライドでDB側権限の話をする)。細部だが目ざとい受講者が気づくと盛り上がる
-->

## ガードレール(1):権限とクエリコスト

| ガードレール | 実装 | 防ぐ事故 |
|---|---|---|
| **読み取り専用** | DB側:読取専用ユーザー(`GRANT MATCH`のみ)/ サーバ側:writeツールを無効化 | データ破壊・汚染 |
| **クエリタイムアウト** | `db.transaction.timeout`(例:10秒) | 暴走クエリでDB占有 |
| **結果サイズ制限** | ツール側で `LIMIT` 強制・行数上限 | コンテキスト溢れ・情報漏洩 |
| **スキーマ露出の制御** | 見せるラベル/プロパティを絞る | 機微データへの誘導 |

> 原則:**プロンプトのお願いではなく、DB・サーバ側で強制する**
> (LLMは「読み取りだけにしてね」を破ることがある)

<!--
- 本章で最も実務価値が高いスライドのひとつ。「デモは動く、本番はここで決まる」
- 二重防御の考え方:ツールdescriptionで誘導(ソフト)+DB権限・サーバ実装で強制(ハード)。ソフトだけの構成は事故る
- 読み取り専用ユーザーはRDBMS運用と同じ発想なので受講者はすぐ腹落ちする。「エージェントは新入社員アカウントと同じ扱いで」という例え
- スキーマ露出の制御は見落とされがち:給与プロパティがスキーマに見えるだけで、エージェントはそれを使うクエリを書き得る
- プロンプトインジェクション(文書内の指示でエージェントを操る)もここで一言触れておく
-->

---

## ガードレール(2):探索の発散防止

エージェントは放っておくと**調べ続けてしまう**:

| 上限 | 例 | 超えたら |
|---|---|---|
| **ステップ上限** | ツール呼び出し 最大10回 | 途中結果で回答を合成 |
| **コスト上限** | 累計トークン 50k / 1タスク | 〃 |
| **時間上限** | 60秒 | 〃 |
| **同一クエリ検出** | 同じクエリ2回で警告注入 | ループから脱出させる |

- 打ち切り時の応答設計が重要:
  **「ここまでで分かったこと+未確認の点」を明示して返す**
- 探索ログ(全ツール呼び出しの記録)は必ず残す → 演習で分析

<!--
- 「発散」の実例を話す:空結果→条件変更→また空→…の無限徘徊、同じクエリを言い回しだけ変えて繰り返す、など。実際にやると誰でも一度は目撃する
- 上限値はタスクの複雑さで調整。マルチホップ調査なら10ステップ程度が目安、と経験値を共有
- 「打ち切り=失敗」にしない設計が肝。部分的な調査結果でも根拠付きで返せば業務価値がある。黙ってエラーにするのが最悪
- 探索ログはデバッグ・コスト分析・改善の全ての起点。「ログのないエージェントは運用できない」と強調して演習に繋ぐ
-->

## 信頼性設計:リトライと自己修正ループ

```
クエリ実行
  ├─ 構文エラー → エラーメッセージをそのままLLMに返す
  │                → LLMが修正して再実行(上限3回)
  ├─ 空結果     → スキーマ再確認/条件緩和/表記ゆれ探索
  └─ 成功       → 次のステップへ
```

- **エラーメッセージは貴重な学習材料**:握りつぶさずLLMに見せる
  - `Unknown relationship type ASSIGNED` → LLMが `ASSIGNED_TO` に自己修正
- 修正上限を必ず設ける(3回失敗したら方針転換 or 打ち切り)
- 頻発する失敗はログから拾い、**ツールdescriptionやスキーマ記述の改善**へ

<!--
- 「エラーを見せれば直る」がLLMエージェントの面白いところ。人間の開発者と同じで、エラーメッセージがあれば自己修正できる
- ありがちな実装ミス:例外をcatchして「エラーが発生しました」だけ返す → LLMは何を直せばいいか分からず同じクエリを繰り返す
- 空結果は「エラーではないが失敗」。構文エラーより対処が難しい。スキーマ再確認に誘導するのが定石
- 最終行が改善サイクルの本質:個々のリトライで凌ぐだけでなく、失敗パターンをシステム側の恒久対策(description改善・スキーマの説明追加)に還元する
- 9章の「クエリ検証」との違いを聞かれたら:検証は実行前の静的チェック、自己修正は実行後のフィードバックループ。併用する
-->

## KGへの書き戻し:自己成長するナレッジベース

エージェントが調査中に得た**新事実**をKGに追記する:

```
エージェント:「調査の過程で判明:顧客Z社もサービスPを利用」
   │
   ▼ 直接書かない!
[提案キュー] 追加案:(Z社)-[:USES]->(サービスP)+根拠(出典・確信度)
   │
   ▼ 人間がレビュー(承認 / 却下 / 修正)
```

## マルチエージェント構成:分業で品質を上げる

| エージェント | 役割 | 主なツール | 書込権限 |
|---|---|---|---|
| **抽出** | 文書→トリプル案の生成(7章の自動化) | 文書読取、スキーマ取得 | 提案キューのみ |
| **検証** | スキーマ適合・既存データとの矛盾チェック(8章) | スキーマ取得、読取クエリ | 承認/却下フラグ |
| **QA** | ユーザーの調査質問に回答(本章前半) | スキーマ取得、読取クエリ | なし |

- 分業の利点:**役割ごとに最適なモデル・プロンプト・権限**を設定できる
  - 抽出=小型モデルで大量処理、検証=大型モデルで精査(8章のコスト戦略)
- 構成しすぎ注意:**まず1エージェントで、ボトルネックが見えてから分割**

<!--
- 7章(抽出)・8章(品質)・本章(QA)で学んだ機能が、それぞれエージェントとして分業される構図。コースの積み上げがここで一枚の絵になる
- 権限列に注目させる:QAエージェントは書込権限ゼロ、抽出も提案止まり。最小権限の原則がマルチエージェントでも貫かれる
- 検証エージェントの実例:「新トリプル(Z社)-[:USES]->(P)は、既存の(Z社)-[:CONTRACT_ENDED]->(P) と矛盾しないか?」を読取クエリで確認する
- 最終行を強く言う:マルチエージェントは流行りだが、分業はオーバーヘッド(通信・デバッグ困難)を伴う。「必要になってから分ける」がコースの一貫した姿勢

## 【ハンズオン】マルチホップ調査をエージェントが解く

質問:**「障害INC-F12の影響を受ける顧客と担当営業は?」**

| Step | エージェントの行動 | 結果 |
|---|---|---|
| 1 | `get_graph_schema` | `(Incident)-[:AFFECTS]->(System)-[:HOSTS]->(Service)` `(Customer)-[:USES]->(Service)` `(Rep)-[:IN_CHARGE_OF]->(Customer)` |

| 5 | 回答合成:**根拠パス付きの一覧+未確認事項** | 完了 |

## 【演習】調査エージェントの実装と探索ログ分析

配布のひな形(MCP接続済み)をベースに:

1. **実装**:2点セットのツールで調査エージェントを構成し、ガードレール(ステップ上限10・タイムアウト・読取専用)を設定
2. **実行**:マルチホップ調査質問を3問投げる(例:「A社に関係する進行中プロジェクトと担当者は?」)
3. **ログ分析**:探索ログを読み、次を記録
   - ステップ数 / 失敗クエリと自己修正の有無 / 無駄な探索
4. **改善**:ツールdescription か上限値を1箇所変えて再実行し、変化を比較
5. 共有:最も効いた改善は何か?

⏱ 45分(実装15分 + 実行10分 + 分析10分 + 改善・共有10分)

<!--
- 演習の主役は4の「改善」。エージェント開発の実態は「ログを読んで説明文を直す」の反復であり、それを1周体験させる
- ありがちなつまずき:MCP接続エラー(環境変数)、ステップ上限が小さすぎて打ち切り多発 → 机間巡視でこの2つを重点ケア
- ログ分析の観点を明示的に教える:「人間ならこの探索をしたか?」と比べると無駄が見つけやすい
- 改善例が出ないグループへのヒント:descriptionに「まず存在確認から始めること」を足す、スキーマ説明にプロパティの値例を足す、など
- 共有では「上限を上げたら精度が上がったがコストも増えた」という報告を拾い、11章のROI・コスト設計に繋ぐ

## 第10章まとめ

- エージェント=**計画→実行→観察のループ**で複数ステップの調査を解く
  - 固定パイプライン(9章)で足りるタスクにはエージェントを使わない
- KGのツール化は**スキーマ取得+クエリ実行の2点セット**が基本形、接続は**MCP**で標準化
- 逐次探索は「完璧な1本」を**修正可能なループ**に置き換える
- 本番の条件:**ガードレール**(読取専用・コスト/ステップ上限)+**自己修正**+**探索ログ**
- 書き戻しは**人間承認フロー付き**で — 自己成長と品質維持の両立

**次章(最終章)**:総合実践 — 要件からDB選定・設計・評価・ROIまで、「効果が出る」ソリューションの意思決定を一気通貫で行う
