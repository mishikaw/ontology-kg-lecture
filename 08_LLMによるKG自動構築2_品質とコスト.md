## 7章のパイプラインに残された問題

7章の抽出結果をそのまま投入すると、こうなる:

```
(トヨタ)──[製造]→(プリウス)
(トヨタ自動車)──[本社所在地]→(豊田市)
(Toyota Motor)──[提携]→(スバル)
```

- **同一企業が3ノードに分裂** → 「トヨタの情報を全部見せて」が答えられない
- 抽出ミス・幻覚トリプルが**検出されないまま混入**
- 全文書を再処理すると**毎回フルコスト**が発生

| 7章で作ったもの | 8章で足すもの |
|---|---|
| 抽出パイプライン(動く) | 名寄せ・品質評価・コスト管理(使える) |

## エンティティ解決(名寄せ)とは

**表記が異なる同一実体を、1つのノードに統合する処理**

```
入力: 「トヨタ」「トヨタ自動車」「Toyota Motor」「トヨタ自動車(株)」
出力: (c:Company {name: "トヨタ自動車", aliases: [...]}) ← 1ノード
```

- 別名・略称・言語違い・法人格表記・旧社名…原因は多様
- RDBの世界の「名寄せ」「マスタ統合」と同じ問題
  - 違い:LLM抽出では**入力が毎回ゆらぐ**ため、発生頻度が桁違い
- 判断が難しい例:
  - 「トヨタ」と「トヨタ紡織」→ **別実体**(前方一致では誤る)
  - 「Meta」と「Facebook」→ **同一実体**(文字列は全く違う)

<!--
- 「トヨタ紡織」「Meta/Facebook」の2例が本質:文字列類似だけでは過剰統合と統合漏れの両方が起きる
- 過剰統合(別物をくっつける)の方が復旧が難しいことを強調。分割は後からできるが、混ざったプロパティの分離は困難
- 受講者への問いかけ:「顧客マスタの名寄せをやったことがある人?あの苦労がLLM抽出では毎日発生します」
- ここから3つの手法を順に見て、最後に組み合わせる流れを予告
-->

## 手法1:ルールベース(正規化+辞書)

```python
import re, unicodedata

def normalize(name: str) -> str:
    name = unicodedata.normalize("NFKC", name)   # 全角/半角統一
    name = re.sub(r"(株式会社|\(株\)|㈱|Inc\.?|Corp\.?)", "", name)
    return name.strip().lower()

ALIAS_DICT = {"トヨタ": "トヨタ自動車",
              "toyota motor": "トヨタ自動車"}    # 人手管理の別名辞書
```

| 項目 | 評価 |
|---|---|
| 精度 | ルールに載る範囲では**ほぼ100%**、載らないものは0% |
| コスト | **ゼロ**(LLM呼び出し不要)、超高速 |
| 弱点 | 辞書メンテが必要、「Meta=Facebook」は書かないと取れない |

<!--
- 「地味だが最初にやるべき手法」。正規化だけで重複の5〜7割が消えるケースは多い
- NFKC正規化・法人格除去は日本語データでは必須の前処理。全角括弧・㈱など実データの汚さを例示する
- 別名辞書はドメイン知識の結晶。5章のオントロジー(用語正規化辞書)がここで効いてくると接続
- 限界の確認:辞書は「既知のゆれ」しか救えない。未知のゆれを次のembeddingで拾う
-->

## 手法2:embedding類似度

```python
# 名称+文脈をembedding化し、近いノード候補を探す
emb = embed("トヨタ自動車(自動車メーカー、愛知県)")
candidates = vector_index.query(emb, top_k=5)
# → [("Toyota Motor", 0.93), ("トヨタ紡織", 0.82), ...]
```

- 文字列が違っても**意味が近ければ拾える**(多言語・略称に強い)
- 名称だけでなく**周辺情報(業種・所在地)を含めて**embedding化すると精度向上
- 弱点:
  - 「トヨタ紡織」のような**近縁の別実体**も高スコアになる
  - **閾値の決定が難しい**:0.9で切る?0.85?→ ドメイン依存
- 実務では「候補生成」に使い、**確定判断は別の手段**で行う

<!--
- 4章で学んだNeo4jベクトルインデックスがここで再登場。技術は同じ、用途が「検索」から「名寄せ候補生成」に変わっただけ
- スコア0.93と0.82の差は僅かに見えるが片方は同一・片方は別実体。「閾値一本で自動判定」の危うさを数字で見せる
- グレーゾーン(0.8〜0.9など)をどうするかが次のLLM判定への橋渡し
- embeddingコストはLLM生成より1〜2桁安いことにも触れる(コスト設計の伏線)
-->

## 手法3:LLM判定

```text
以下の2つのエンティティは同一の実体ですか?
A: 名称「トヨタ」/ 文脈「...プリウスを製造する大手メーカー...」
B: 名称「Toyota Motor」/ 文脈「...Japanese automaker...」
JSONで回答: {"same_entity": true/false, "confidence": 0-1, "reason": "..."}
```

| 項目 | 評価 |
|---|---|
| 精度 | **最高**。文脈・世界知識を使える(Meta=Facebookも判定可) |
| コスト | **最高**。1判定=1API呼び出し |
| 落とし穴 | 全ペア比較は**O(N²)** — 1万ノードなら5,000万ペアで破綻 |

→ **候補を絞ってから**LLMに聞くのが鉄則

<!--
- LLMの強み:「トヨタ紡織はトヨタグループだが別法人」という世界知識を使った判定ができる
- O(N²)の恐怖を具体的な数字で:1万ノード全ペア≒5,000万回のAPI呼び出し。1回0.1円でも500万円
- だからembedding候補生成(ブロッキング)が前段に必須。従来の名寄せでも「ブロッキング→詳細比較」は定石で、LLMはその詳細比較部分の置き換え
- confidenceとreasonを出させるのは後の人間レビュー(プロビナンス設計)への布石
-->

## 3手法の比較と組み合わせ戦略

| | ルールベース | embedding類似度 | LLM判定 |
|---|---|---|---|
| 精度 | 辞書内なら◎ | ○(閾値依存) | ◎ |
| コスト/件 | ほぼ0円 | 低(embedding) | 高(生成API) |
| Meta=Facebook | ✕(辞書次第) | △ | ◎ |
| トヨタ紡織の誤統合 | ○(防げる) | ✕(危険) | ◎ |

**実務の定石:ファネル(じょうご)型パイプライン**

```
全エンティティ
 → ①正規化・辞書で確定統合(大多数を0円で処理)
 → ②embeddingで候補ペア生成(top-k、閾値でグレーゾーン抽出)
 → ③グレーゾーンだけLLM判定(全体の数%まで絞る)
 → ④低confidenceは人間レビューキューへ
```

<!--
- 本章前半の結論スライド。「どれを使うか」ではなく「どう並べるか」が答え
- ファネルの各段で件数が減るイメージを口頭で:10,000件→正規化で7,000件に統合→候補ペア2,000→LLM判定300→人間レビュー30
- コスト構造:高い手法ほど後ろに置き、通る件数を減らす。これは後半のモデル段階戦略と同じ思想
- RDB対比:「インデックスで絞ってから精査する実行計画と同じ発想」と言うと腹落ちする
-->

## 外部KGへのリンキング:安定識別子を持つ

6章の応用:**自社ノードをWikidata等の外部IDに紐付ける**

```cypher
MERGE (c:Company {name: "トヨタ自動車"})
SET c.wikidata_id = "Q53268",       // 世界共通の安定ID
    c.aliases = ["トヨタ", "Toyota Motor", "TMC"]
```

- **表記ではなくIDで同一性を管理**:「Q53268 = トヨタ自動車」は不変
- 効能:
  - 名寄せの**確定的な根拠**になる(ID一致=同一実体)
  - 外部KGから属性を補完(業種、本社、子会社…6章のSPARQL)
  - 複数システム・複数KG間の**共通キー**になる
- 社内エンティティ(自社製品・部署)には**自前の安定ID体系**を用意する

<!--
- 6章で「外部知識の取り込み手段」として学んだWikidataが、ここでは「IDレジストリ」として効く
- RDB対比:「natural key(社名)ではなくsurrogate key(ID)で管理する」という設計原則そのもの
- エンティティリンキングの手順:名称+文脈でWikidata検索→候補をLLMで曖昧性解消(Apple社 vs りんご)
- 社内固有エンティティはWikidataに無い。だから「公開エンティティは外部ID、社内エンティティは自前ID」の二本立てが現実解
- 全ノードにリンキングする必要はない。ハブになる重要エンティティ(企業・人・製品)から優先的に
-->

## 品質評価:precision / recall の測り方

**抽出トリプルをゴールドデータ(正解)と突き合わせる**

| 指標 | 定義 | 低いとき何が悪いか |
|---|---|---|
| **Precision** | 抽出した中で正しい割合 | 幻覚・誤抽出がグラフを汚す |
| **Recall** | 正解のうち抽出できた割合 | 抽出漏れ=答えられない質問が増える |

計算例(ある文書で正解トリプル20件、抽出25件、うち正解18件):

```
Precision = 18 / 25 = 0.72   ← 7件はゴミ(幻覚・誤り)
Recall    = 18 / 20 = 0.90   ← 2件は取り漏らし
F1        = 2·(0.72·0.90)/(0.72+0.90) = 0.80
```

- 判定は「トリプル一致」:主語・述語・目的語がすべて一致(**名寄せ後に**比較)
- 用途で優先度が変わる:**監査用途→precision優先、探索用途→recall優先**

<!--
- 「測っていないものは改善できない」。プロンプトをいじって"良くなった気がする"で回すのがデモ止まりの典型
- トリプル一致の落とし穴:名寄せ前に比較すると「トヨタ」vs「トヨタ自動車」で不一致扱いになる。評価にも名寄せが必要という依存関係を指摘
- 部分一致(主語・目的語は合うが関係型が違う)をどう数えるかはチームで規約を決める。厳密一致+緩和一致の2本立てで測ると診断しやすい
- 受講者への問いかけ:「自社ユースケースはprecision優先?recall優先?」— 9章・11章の設計判断に繋がる
-->

## 評価用ゴールドデータの作り方

**全文書は不要。少数を丁寧に。**

| ステップ | 内容 | 目安 |
|---|---|---|
| 1. サンプリング | 文書タイプ・長さが偏らないよう層化抽出 | 20〜50文書 |
| 2. LLMでドラフト | 7章のパイプラインで抽出(ゼロから書かない) | — |
| 3. 人手で修正 | 誤りを直し、漏れを足す → これが正解データ | 1文書10〜20分 |
| 4. ダブルチェック | 2人目が確認、判断が割れたら基準を文書化 | 重要分のみ |

- ポイント:**修正の記録自体が資産**(どんな誤りが多いか=改善の方向)
- アノテーション基準書を書く:「子会社の製品は親会社に紐付ける?」等の判断ルール
- スキーマ(5章)が変わったらゴールドデータも改訂 — バージョン管理する

<!--
- 「正解データを人手で作る」と聞くと引かれるが、20文書×15分=5時間で永続的な評価基盤ができると考えれば安い投資
- 「LLMドラフト→人手修正」方式はゼロから書くより3〜5倍速い。ただしLLMの誤りに引きずられるバイアスに注意(だからダブルチェック)
- 判断が割れる例を1つ実演すると良い:「『トヨタはEV開発を加速』→ (トヨタ)-[開発]->(EV) は正解に含める?粒度の規約が要る」
- Gitでバージョン管理し、パイプライン変更のたびにCIで自動評価する運用まで話すと実務イメージが固まる
-->

## LLM-as-a-judge:活用と限界

**ゴールドデータが無い範囲を、別のLLMに採点させる**

```text
原文: 「トヨタ自動車は2025年、スバルとEV開発で提携拡大した」
抽出: (トヨタ自動車)-[提携]->(スバル)
質問: この抽出は原文から正しく導けますか? {"verdict": ..., "reason": ...}
```

| 使いどころ | 限界 |
|---|---|
| 全件に薄く網をかける一次スクリーニング | judge自身も誤る(**判定の幻覚**) |
| precision系の検証(原文と突き合わせ) | recall系(漏れ検出)は苦手 |
| 人間レビューの優先順位付け | 自己採点バイアス(同系モデルに甘い) |

> 鉄則:**judgeの判定をゴールドデータで校正してから使う**
> (人間の判断との一致率を測る — judgeの精度も測定対象)

<!--
- 位置づけ:ゴールドデータ(狭く深く)とjudge(広く浅く)の併用。judgeはゴールドの代替ではない
- 「judgeの判定は原文がプロンプトに入っているから幻覚しにくい」が、それでも誤る。judge自体の精度をゴールドで測る(メタ評価)が抜けがち
- recallに弱い理由:「漏れ」は原文全体を読んで気づく必要があり、1トリプルずつの検証では見えない
- 抽出と同じモデルでjudgeすると誤りに共通の癖が出て甘くなる。別系統のモデルを使うのが定石
- コスト面:judgeも生成APIなので全件・毎回は高い。サンプリング+変更時に限定する
-->

## ハルシネーション対策:プロビナンス(原文根拠)

**すべてのエッジに「どの文書のどの文から来たか」を持たせる**

```cypher
MERGE (a:Company {name: "トヨタ自動車"})
MERGE (b:Company {name: "スバル"})
MERGE (a)-[r:提携]->(b)
SET r.source_doc  = "news_2025_0412.txt",
    r.source_text = "トヨタ自動車はスバルとEV開発で提携拡大…",
    r.confidence  = 0.92,
    r.extracted_at = datetime(), r.model = "claude-haiku-4-5"
```

- **根拠を辿れないトリプルは存在しない**状態を作る(投入時に必須化)
- 9章のGraphRAGで「回答の根拠として原文を提示」できる — 1章デモの根拠表示の正体
- 誤りが見つかったら `source_doc` から**同一ソース由来を一括調査**できる

<!--
- 本章で最も実務価値の高いスライドのひとつ。プロビナンスは「後から足せない」ので最初から設計に入れる
- 1章デモで見た「根拠パス付き回答」はこの仕組みで実現されていた、と種明かしを予告(9章で完結)
- RDB対比:「監査ログ・リネージのカラムを持つのと同じ。ただしグラフでは関係(エッジ)そのものに持てるのが利点」— 3章の「関係に属性」の実例
- model・extracted_atを残す理由:モデル更新時に「旧モデル抽出分だけ再検証」が可能になる(継続更新の伏線)
- source_textの容量が気になる場合はオフセット(文書ID+文字位置)参照にする設計もあると補足
-->

## 確信度スコアと人間レビューフロー

**全件人手は不可能 → 確信度で仕分けて人間を最小限に使う**

```
confidence ≥ 0.9  → 自動承認(そのまま本番グラフへ)
0.6 ≤ conf < 0.9  → レビューキューへ(:PendingReview ラベル)
confidence < 0.6  → 破棄 or 再抽出
```

```cypher
// レビュー画面用:保留中トリプルを根拠付きで一覧
MATCH (a)-[r {status: "pending"}]->(b)
RETURN a.name, type(r), b.name, r.source_text, r.confidence
ORDER BY r.confidence ASC LIMIT 50
```

- レビュー結果(承認/却下)を記録 → **few-shot例・辞書の改善材料**に還流
- 閾値は固定しない:precision実測を見ながら調整する運用

<!--
- 「人間レビュー=全件チェック」ではない。機械が仕分け、人間はグレーゾーンだけ見る。ファネル思想の再登場
- confidenceの出所:LLMに自己申告させる値は過信気味の傾向がある。judge判定・複数回抽出の一致率と組み合わせると信頼性が上がる
- レビューUIは凝らなくてよい。最初はNeo4j Browserやスプレッドシートで十分。回る仕組みを先に作る
- 却下されたトリプルは消すだけでなく「なぜ却下か」を記録 → プロンプト改善のfew-shot負例になる好循環を強調

## グラフ品質の検証:Cypherでチェックする

**スキーマ違反・構造異常を定期検出(品質ダッシュボードの中身)**

```cypher
// ① スキーマ違反:許可していないラベル・関係型(5章のオントロジーと突合)
MATCH (n) WHERE NOT any(l IN labels(n) WHERE l IN $allowed_labels)
RETURN labels(n), count(*)

// ② 孤立ノード:どこにも繋がらない(抽出断片・名寄せ漏れの疑い)
MATCH (n) WHERE NOT (n)--() RETURN labels(n), count(n)

// ③ 重複エッジ:同一ペア間に同型の関係が複数
MATCH (a)-[r]->(b) WITH a, type(r) AS t, b, count(r) AS c
WHERE c > 1 RETURN a.name, t, b.name, c ORDER BY c DESC
```

- 週次バッチで実行し、件数の**推移**を記録 → 悪化=パイプライン異常のシグナル
- 制約(4章)で防げるものは制約で:`CREATE CONSTRAINT ... IS UNIQUE`

<!--
- RDB対比:「データ品質チェックSQL(NULL率・参照整合性チェック)のグラフ版」。発想は完全に同じ
- ①は5章のオントロジーが検証仕様書として機能する場面。「スキーマは抽出の制約であり検査の基準でもある」
- 孤立ノードは必ずしも悪ではない(名寄せ待ちの新エンティティかも)。「疑いリスト」として扱う
- 重複エッジは7章のチャンク重複・再実行が主因。MERGEの使い方(4章)で予防できるがゼロにはならないので検出も要る
- 「絶対値より推移」が運用の勘所。先週100件→今週3,000件なら上流で何かが壊れている
-->

## コスト設計:トークン量を見積もる

**実装前に電卓で概算できる — 例:社内文書10,000件のKG構築**

```
入力: 10,000文書 × 平均4,000トークン        = 40M tokens
出力: 1文書あたり抽出JSON 約800トークン      =  8M tokens

小型モデル(例: $1/M入力, $5/M出力)で抽出:
  40M × $1 + 8M × $5 = $40 + $40 = $80

大型モデル(例: $3/M入力, $15/M出力)で全件抽出すると:
  40M × $3 + 8M × $15 = $120 + $120 = $240(3倍)
```

- 支配項は**入力トークン×文書数**。チャンク重複(7章)は入力を10〜20%増やす
- 名寄せLLM判定・judge検証・再実行のコストも忘れず加算
- **1文書あたり単価**を出しておくと増分更新(月次○件)の運用費が即算できる

<!--
- 「見積もれない技術は稟議が通らない」。この電卓芸ができるだけで社内での説得力が段違い
- 単価は変動するので考え方だけ持ち帰らせる:文書数×平均トークン×単価、支配項はどれか
- 隠れコストを列挙:失敗リトライ、プロンプト改善のための再実行(全件×N回!)、評価用judge。再実行が一番効くので7章の冪等設計・差分処理が金額に直結する
- 受講者への問いかけ:「自社の対象文書は何件・平均何文字?」— その場で概算させると自分ごとになる
- ベクトルRAGのembeddingコストとの比較:KG構築は生成APIを使うぶん1〜2桁高い。だからこそ9章の「使い分け」が重要という伏線
-->

## コスト削減:段階的モデル戦略・キャッシュ・バッチ

| 戦略 | 内容 | 削減効果の目安 |
|---|---|---|
| **段階的モデル** | 抽出=小型モデル、検証・難所だけ大型モデル | 生成費 50〜70%減 |
| **プロンプトキャッシュ** | スキーマ+few-shot(毎回同じ部分)をキャッシュ | 入力費 大幅減 |
| **バッチAPI** | 即時性不要な構築ジョブを非同期実行 | 約50%減 |
| **LLMを使わない** | 構造化データはCSV直接投入(7章) | その分100%減 |

```
定石:安いものから順に試し、高いものは通す件数を絞る
  正規化(0円)→ embedding(低)→ 小型LLM(中)→ 大型LLM(高)→ 人間(最高)
```

- KG構築は**バッチ処理**なのでバッチAPIと相性が良い(リアルタイム不要)
- 段階戦略の判断基準:小型モデルのprecision/recallを**測ってから**決める(前半の評価が効く)

<!--
- 名寄せファネルと完全に同じ思想:「高い資源ほど後ろに置き、通す量を絞る」。本章を貫く原理として明示的に回収する
- プロンプトキャッシュが効く理由:抽出プロンプトは「スキーマ+few-shot(固定・長い)+文書(可変)」構造。固定部が入力の大半を占めることが多い
- 「小型で十分か」は感覚で決めない。ゴールドデータで小型/大型のF1を比較し、差額と精度差を並べて判断する — 評価基盤があるからできる意思決定
- 人間を最高コスト資源として同じ表に載せるのがポイント。レビューフローも「人間の通過量を絞る」設計だった
- バッチAPIの注意:結果到着まで時間がかかるので、パイプラインを非同期前提で組む(7章の冪等性設計が活きる)
-->

## 継続更新:KGは作って終わりではない

| 課題 | 設計 |
|---|---|
| **増分処理** | 新規・変更文書のみ抽出(ハッシュ・更新日時で検知)→ 既存グラフにMERGE |
| **名寄せの継続** | 新ノードは既存ノードとの照合を必ず通す(ファネル再利用) |
| **経時変化** | 「社長」「所在地」は変わる → エッジに `valid_from` / `valid_to` |
| **鮮度管理** | ノード・エッジに `last_verified` → 古い事実を検知・再確認 |

```cypher
// 1年以上未検証の重要エンティティを洗い出す
MATCH (c:Company)-[r]->()
WHERE r.last_verified < datetime() - duration('P1Y')
RETURN c.name, type(r), r.last_verified ORDER BY r.last_verified
```

- 「事実が変わる」のは誤りではない — **時間軸を持たない設計が誤り**
- 鮮度SLA(例:企業情報は90日以内に再検証)は11章の運用設計で扱う

<!--
- 「KGの価値は鮮度に比例する」。古いKGに基づくGraphRAGの回答は自信満々の過去情報=一種のハルシネーション
- 増分処理は7章の冪等性・差分更新設計の回収。あの設計がないと毎月フルビルド費用がかかる、とコストの話に接続
- valid_from/valid_toはRDBの履歴テーブル(タイプ2 SCD)と同じ発想。DWH経験者には「ディメンションの緩やかな変化」と言えば通じる
- どこまで時間軸を持つかは要件次第。全エッジに持つと複雑化するので「変わりうる関係」に限定するのが現実解
- 演習説明の前に一呼吸:「ここまでで名寄せ・品質・コスト・更新の4点セットが揃った。次はこれを自分の手で」
-->

## 第8章まとめ

- **名寄せはファネルで**:正規化・辞書 → embedding候補 → LLM判定 → 人間レビュー
  - 外部ID(Wikidata等)による**識別子ベースの同一性管理**が最終形
- **測れないものは改善できない**:少数精鋭のゴールドデータでP/Rを常時計測、judgeは校正してから補助に
- **根拠のないトリプルを入れない**:プロビナンス+確信度+レビューキューをエッジ設計に最初から組み込む
- **コストは電卓で見積もる**:支配項は入力トークン×文書数。安い資源から順に使い、高い資源は通過量を絞る
- KGは**生き物**:増分処理・時間軸・鮮度管理まで設計して初めて「使えるシステム」
