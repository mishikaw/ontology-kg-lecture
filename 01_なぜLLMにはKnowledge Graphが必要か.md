## みなさんへの質問

LLMアプリ(チャットボット、RAG)を作ったことがある方へ:

- 「**それっぽいが間違った回答**」に悩まされたことは?
- 「**この回答の根拠は?**」と聞かれて困ったことは?
- 「**複数の情報を辿らないと答えられない質問**」で精度が落ちたことは?

→ 本コースはこの3つを解決する技術を学ぶ

<!--

## LLM実務の3大課題

| 課題 | 症状 | ビジネスへの影響 |
|---|---|---|
| **ハルシネーション** | 事実と異なる内容を自信満々に生成 | 誤情報による意思決定ミス、信頼失墜 |
| **根拠提示の欠如** | 「なぜその回答か」を示せない | 監査・説明責任が必要な業務で使えない |
| **マルチホップ推論の弱さ** | 複数の事実を繋いだ推論で精度低下 | 調査・分析業務での実用性が低い |

- LLM単体の性能向上では**構造的に解決しない**
- 外部知識との接続 = RAG(Retrieval-Augmented Generation)が標準解
- しかし**ベクトルRAGだけでは不十分** ← 今日の主題

<!--
- 「モデルが賢くなれば解決する」という期待に釘を刺す:知識は常に更新される・社内知識は学習されていない
- RAGは前提知識として軽く復習(受講者は経験ありの想定だが復習で揃える)
- 「ベクトルRAGで十分なケースも多い」ことも公平に言っておく。9章・11章で使い分けを学ぶ
-->

## 復習:ベクトルRAGの仕組み

```
文書群 → チャンク分割 → embedding化 → ベクトルDB
                                          ↑
質問 → embedding化 → 類似度検索 ──────────┘
     → 上位k件をプロンプトに注入 → LLMが回答生成
```

- 質問と**意味的に近いテキスト片**を探してLLMに渡す
- 得意:単一チャンクに答えが書いてある質問(FAQ、文書検索)
- 前提:**答えがどこかのチャンクにまとまって書かれている**

<!--
- 受講者は実装経験がある想定なので1枚でさらっと復習
- 強調すべきは最後の「前提」:この前提が崩れる質問類型が次のデモ
- チャンク分割の時点で「文書をまたぐ関係」の情報が捨てられている、と図を指しながら指摘
-->

## 【デモ1】ベクトルRAGが失敗する質問

サンプル:社内文書100件(人事情報、プロジェクト報告、取引記録)

| 質問タイプ | 例 | ベクトルRAGの結果 |
|---|---|---|
| 単純検索 | 「A社との契約内容は?」 | ✅ 正答 |
| **マルチホップ** | 「Aさんと Bさんの共通の取引先は?」 | ❌ 片方しか見つからない |
| **集約** | 「製造業の顧客は何社ある?」 | ❌ 数え漏れ・誤答 |
| **影響分析** | 「この障害の影響を受ける顧客は?」 | ❌ 依存関係を辿れない |

<!--
- ここは実際にライブで動かす(配布デモ環境。付録A参照)
- 失敗する理由をその場で解説:
  - マルチホップ:AさんとBさんの情報は別チャンクにあり、類似度検索では「共通の取引先」という関係が探せない
  - 集約:類似チャンク上位k件しか取れないので全件数えられない
  - 影響分析:システム→サービス→顧客という依存の連鎖はどのチャンクにも明示的に書かれていない
- 「賢いプロンプト」では直らないことも見せると説得力が増す
-->

## なぜ失敗するのか:チャンクの限界

- **関係が分断される**:「Aさんの取引先」と「Bさんの取引先」は別々の文書
  - 類似度検索は「意味が近いテキスト」を探すだけで、**関係を辿れない**
- **全体を見られない**:上位k件しか取得しない → 集約・網羅が原理的に不可能
- **暗黙の構造が消える**:組織図、システム依存、契約関係…
  - 人間は「構造」を頭に持っているから文書を読み解ける

> 足りないのは「テキストの意味検索」ではなく
> **「エンティティと関係の構造」を保持・探索する仕組み**

<!--
- デモ1の種明かしを一般化する重要スライド
- 「embeddingの精度を上げても解決しない。情報の持ち方の問題」と構造的な限界であることを強調
- 最後の枠が本コースの中心テーゼ。「これがKnowledge Graphです」と次のスライドへ繋ぐ
-->

## Knowledge Graph とは

**エンティティ(ノード)と関係(エッジ)で知識を表現したグラフ**

```
(Aさん:社員)─[担当]→(X社:顧客)←[担当]─(Bさん:社員)
(X社:顧客)─[契約]→(サービスP)─[稼働]→(サーバS)
```

- 「Aさん」「X社」は**ノード**(ラベルとプロパティを持つ)
- 「担当」「契約」は**エッジ**(関係の種類を持つ)
- 質問「AさんとBさんの共通の取引先は?」
  → **グラフのパターンマッチで確実に解ける**

<!--
- 板書的にゆっくり。ノード・エッジという語をここで導入
- さっき失敗した質問がグラフ表現なら「Aさんから辿ってBさんにも辿られる顧客を探す」という機械的な探索で解けることを図で示す
- 「勘の良い人はRDBでもJOINで書けると思ったはず。正解。その比較は3章で徹底的にやります」と予告(RDBMS経験者の疑問を先回り)
-->

## 【デモ2】同じ質問を GraphRAG で

同じ社内文書100件から構築した Knowledge Graph に対して:

| 質問 | ベクトルRAG | GraphRAG |
|---|---|---|
| 「AさんとBさんの共通の取引先は?」 | ❌ | ✅ X社、Y社(根拠パス付き) |
| 「製造業の顧客は何社?」 | ❌ | ✅ 12社(一覧付き) |
| 「この障害の影響を受ける顧客と担当営業は?」 | ❌ | ✅ 依存チェーンを提示 |

- 回答に**根拠(グラフの経路)**が付く → 説明可能性
- 存在しない関係は「見つからない」と答える → ハルシネーション抑制

<!--
- ライブデモ第2弾。ここがこの章のクライマックス
- 中身(どうやってKGを作ったか、どうやって検索しているか)はブラックボックスでよいと明言。「これを全部自分で作れるようになるのがこのコース」
- 根拠パスの表示を必ず見せる:「この回答はこの経路から来た」が出ること自体が業務価値
- 3大課題(ハルシネーション・根拠・マルチホップ)がそれぞれ改善されていることを対応づけて振り返る

## 用語の交通整理

```
┌─ 概念レイヤ ──────────────────────────────┐
│ Ontology:概念と関係の設計図(スキーマ)   │ ← 5章
├─ データレイヤ ────────────────────────────┤
│ Knowledge Graph:実データのグラフ          │ ← 7-8章で構築
│   表現形式: プロパティグラフ / RDF        │ ← 2章 / 6章
├─ 格納レイヤ ──────────────────────────────┤
│ Graph DB:グラフを格納・検索するDB         │ ← 3-4章
│   例: Neo4j(プロパティグラフ)           │
│       トリプルストア(RDF)               │
└───────────────────────────────────────────┘
```

<!--
- 初学者が最も混乱する4用語をレイヤで整理する重要スライド
- RDBに対応づけると腹落ちしやすい:Ontology≒スキーマ定義、KG≒データ本体、GraphDB≒DBMS
- RDFは「表現形式のひとつ」であり、KGはRDFでなくても作れる(本コースはプロパティグラフ主軸)と明言
- 各用語がどの章で登場するかを示し、ロードマップへの期待をつくる

## LLM × KG:3つの組み合わせパターン

| パターン | 方向 | 内容 | 章 |
|---|---|---|---|
| **1. LLMでKGを作る** | LLM → KG | 文書からエンティティ・関係を自動抽出 | 7-8章 |

## 第1章まとめ

- ベクトルRAGは**マルチホップ・集約・影響分析**の質問に構造的に弱い
  - 原因:チャンク化で**関係の構造**が失われるから
- Knowledge Graph = **エンティティと関係**で知識を表現し、構造ごと保持する
- LLM × KG は双方向:**LLMがKGを作り、KGがLLMを強化する**
- 用語のレイヤ:Ontology(設計図)/ KG(データ)/ Graph DB(格納)

**次章**:グラフデータモデルの基礎 — RDBスキーマとの対応でグラフの考え方を身につける
