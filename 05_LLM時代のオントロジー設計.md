## オントロジーとは — 哲学は1枚で終わり

> 「**概念とその関係の明示的な仕様**」(Gruber, 1993 を実務向けに要約)

- 語源は哲学の存在論(ontology)。だが本コースでの扱いは:

| ではなく | として扱う |
|---|---|
| 世界の正しい分類体系 | **このシステムが扱う語彙とルールの合意** |
| 学術的成果物 | **グラフのスキーマ + 用語集 + 制約**(≒ DDL+設計書) |
| 一度作る完成品 | **CQに応えるために育てる道具** |

- RDBエンジニアの語彙で言えば:**ER図 + DDL + 命名規約 + コード表**を1つにしたもの

<!--
- 哲学的背景はこのスライドだけと宣言済み。ここで深入りしない自制を講師自身も守る
- 「明示的な仕様」の対義語は「暗黙の了解」。チームの頭の中にしかない語彙定義を書き下したもの、と言い換える
- RDB対応(ER図+DDL+コード表)が一番伝わる。ただし「閉じ方」が違う点は後のスライドで
- 学術的なオントロジー工学(OWL公理、記述論理)は付録Eに深掘り資料があると案内し、期待値を切っておく
-->

## なぜ今オントロジーか:LLMの実用価値3点

**LLM登場前**:人間のための語彙統一 → 手間の割に効果が見えにくかった
**LLM登場後**:**LLMの入出力を制御する装置**として直接効く

| # | 実用価値 | 効く場所 |
|---|---|---|
| 1 | **抽出のスキーマ制約**:structured outputのJSONスキーマになる | 7章 KG構築 |
| 2 | **Text2Cypherへの入力**:スキーマの質=クエリ生成精度 | 9章 GraphRAG |
| 3 | **用語の正規化辞書**:同義語・表記ゆれを収束させる | 8章 名寄せ |

> スキーマなしでLLMを走らせる = **型なし・バリデーションなしのAPI**を公開するのと同じ

## 実用価値1:structured output のスキーマ制約

同じ文「田中は2020年からアクメ社で働いている」を抽出させると:

<div class="columns">
<div>

**スキーマなし(毎回ブレる)**

```json
{"s":"田中","p":"勤務","o":"アクメ社"}
{"subject":"田中太郎",
 "relation":"works_at", ...}
{"人物":"田中","雇用主":"アクメ"}
```

</div>
<div>

**スキーマあり(常に同型)**

```json
{"entities":[
  {"id":"田中","label":"Person"},
  {"id":"アクメ社","label":"Company"}],
 "relations":[
  {"type":"WORKS_AT","from":"田中",
   "to":"アクメ社","since":2020}]}
```

</div>
</div>

- 関係型を **enum で列挙** → スキーマ外の関係を**構造的に出せない**
- ハルシネーション抑制:自由生成の余地を絞るほど事故が減る

<!--
- 左側は実際にスキーマなしプロンプトを3回叩いた実例(と同等のもの)。「毎回パースコードを書き直すことになる」と実害を語る
- 右側のenum制約は「出力の語彙をこちらが決める」ということ。LLMに命名させない、が鉄則
- structured output(JSONスキーマ強制)機能は主要LLM APIに標準装備。オントロジーはその設定ファイルになる
- 7章の「オントロジー誘導型抽出」の予告:許可する型を絞るほど precision が上がる(recallとのトレードオフは8章)
-->

## 実用価値2:Text2Cypher への入力

質問「佐藤さんの部署のメンバーは?」をCypherに変換させる:

<div class="columns">
<div>

**スキーマ情報なし**

```cypher
-- LLMの想像で書かれる
MATCH (p:Employee
  {employee_name:'佐藤'})
-[:IN_DEPT]->(d)...
```

存在しないラベル・関係を**捏造**

</div>
<div>

**スキーマ情報あり**

```
スキーマ:
(:Person {name})-[:BELONGS_TO]
  ->(:Department {name})
```

```cypher
MATCH (:Person {name:'佐藤'})
  -[:BELONGS_TO]->(d:Department)
  <-[:BELONGS_TO]-(m:Person)
RETURN m.name;
```

</div>
</div>

> **スキーマの質と伝え方が、Text2Cypherの精度をほぼ決める**(9章で実測)

<!--
- 左の失敗は誰もが最初に踏む。LLMは「ありそうなスキーマ」を平気で捏造する — 1章のハルシネーションと同じ現象がクエリ生成でも起きる
- プロンプトに入れるスキーマ表現:ラベル・関係型・プロパティ・方向。関係の説明文(コメント)があるとさらに精度が上がる
- ここでも命名が効く:BELONGS_TO のような自明な動詞名はLLMが正しく選びやすい。暗号的な名前(REL_001)は事故る
- 「オントロジー=ドキュメント」ではなく「オントロジー=プロンプトの部品」という感覚の転換を促す
-->

## 実用価値3:用語の正規化辞書

現場テキストは表記が揺れる:

| 原文の表記 | 正規化先 |
|---|---|
| 「営業部」「営業本部」「Sales」「営業(第一)」 | `(:Department {name:'営業部'})` |
| 「K8s」「Kubernetes」「クバネテス」 | `(:Skill {name:'Kubernetes'})` |
| 「勤務する」「所属」「在籍」「working at」 | `[:WORKS_AT]` |

- オントロジーに**正式名 + 同義語リスト(alias)**を持たせる
- 抽出プロンプトに埋め込む:「以下の正式名に正規化せよ」
- 8章のエンティティ解決(名寄せ)の**参照辞書**になる

<!--
- 「トヨタ/トヨタ自動車/Toyota」問題(8章の主題)の予防線が本章のオントロジー。作る段階で辞書を仕込む
- aliasを持たせる場所:クラス定義のメタデータでも、正規化用のプロンプト部品でもよい。大事なのは一元管理
- 受講者の現場の表記ゆれ実例を聞いてみる(部署名・製品名は鉄板ネタ)
- 正規化を抽出時にやるか、投入後の名寄せでやるかは両方使う。前者で7割潰し、後者で残りを拾うのが実務感覚
-->

## RDBスキーマとの違い:閉世界 vs 開世界

実例:「佐藤さんの部署が登録されていない」とき

| | RDB(**閉世界仮説**) | KG(**開世界仮説**) |
|---|---|---|
| 意味 | 部署は**無い**(またはNOT NULL違反) | 部署を**まだ知らない**だけ |
| 「〜でない」の判断 | 表に無ければ偽 | 記載が無くても**不明** |
| スキーマ外のデータ | INSERTできない(拒否) | いつでも足せる(許容) |

- KGは**知識が常に不完全**である前提で設計する
- LLM回答への影響:「見つからない」≠「存在しない」— 回答文言の設計にも効く(9章)

<!--
- 世界観の違いというと大げさだが、実務では「NOT NULL・外部キーで完全性を守る」というRDBの習慣が通用しない、という話
- 好例:社員マスタは全員分ある(閉じている)が、社員のスキル情報は一部しか無い(開いている)。KGは後者のようなデータが主戦場
- GraphRAGの回答設計への影響は重要:「該当なし」と答えるとき「登録がないだけかもしれません」と添えるかどうかは開世界の理解次第
- 「では制約は一切かけないのか?」という疑問が出るはず → 次のスライドで「足せる自由と守るべき制約のバランス」を扱う
-->

## 開世界でも「育てられる」設計にする

**後から関係を足せるのがKGの強み — ただし無秩序は品質崩壊**

| 変更 | RDB | KG |
|---|---|---|
| 関係の種類を追加 | ALTER TABLE / 中間テーブル追加+移行 | リレーション型を書き足すだけ |
| エンティティ種を追加 | テーブル追加+JOIN書き換え | ラベル追加。既存クエリは無傷 |

守るべき最低限のガードレール:

- **IDの一意性制約**(4章)— これだけは最初から
- **スキーマ定義書を単一ソース化** — 抽出用JSONスキーマもここから生成
- 追加は自由、**改名・削除はレビュー必須**(既存クエリ・プロンプトが壊れる)

<!--
- 3章で扱った「スキーマ変更コスト」の裏返し。KGの進化容易性は本物だが、統制なしだと8章で学ぶ品質問題(重複・スキーマ違反)を自ら生む
- 「単一ソース化」が実務の勘所:スキーマ定義(YAML等)→ 抽出用JSONスキーマ / Text2Cypher用プロンプト / ドキュメントを生成する構成が理想
- 改名の怖さ:WORKS_AT を WORKS_FOR に変えた瞬間、既存データとの二重管理が始まる。RDBのカラム改名より発見が遅れがち
- 「まず小さく作って育てる」は11章のスモールスタート戦略に直結すると予告
-->

## 設計手順:コンピテンシークエスチョン(CQ)駆動

> **CQ = このKGが答えられるべき具体的な質問のリスト**

```
Step 1  CQを列挙する(5〜10個、実際の業務の質問文で)
Step 2  CQの名詞 → エンティティ候補 / 動詞 → 関係候補を抽出
Step 3  粒度と命名を決める(ノードか、プロパティか)
Step 4  制約を付ける(ID、domain/range、カーディナリティ)
Step 5  各CQがスキーマ上で「辿れるか」を検証 → 辿れなければ修正
```

- 「正しい分類」からではなく「**答えたい質問**」から始める
- CQは受入テストになる:KG完成後、CQが解ければ合格

<!--
- 本章で一番持ち帰ってほしい手順。「何をノードにするか迷う」問題への再現可能な答え
- CQの質が命:「うちの組織について何でも」はCQではない。「障害Xの影響顧客は?」のように固有名詞を入れた質問文で書かせる
- Step 5 が形骸化しやすい:紙の上でパターン (a)-[:R]->(b) を指でなぞって辿れるか確認するだけでも設計ミスの大半が見つかる
- CQ=受入テストという発想はエンジニアに馴染む。11章のROI測定でも「CQ正答率」が指標になると予告
-->

## CQ→スキーマ:ステップバイステップ実例

**CQ**:「この**障害**の影響を受ける**顧客**と**担当営業**は?」

```
Step 2  名詞:障害 / システム / サービス / 顧客 / 営業(Person)
        動詞:影響する / 依存する / 契約する / 担当する

Step 3-4  スキーマ化:
  (:Incident)-[:AFFECTS]->(:System)
  (:Service)-[:DEPENDS_ON]->(:System)
  (:Customer)-[:SUBSCRIBES]->(:Service)
  (:Person)-[:ACCOUNT_OF]->(:Customer)

Step 5  検証:障害→AFFECTS→System←DEPENDS_ON←Service
        ←SUBSCRIBES←Customer←ACCOUNT_OF←Person ✅ 辿れる
```

<!--
- 1章のデモ質問の種明かしがここで完了する:「あの影響分析はこのスキーマ設計があったから答えられた」
- Step 2 で「システム」はCQの文面に無いが暗黙に必要、という点に注目させる:CQを実データで解こうとすると中間概念が浮かび上がる
- Step 5 の検証はそのまま4章の可変長パスのクエリになる:MATCH (i:Incident)-[:AFFECTS|DEPENDS_ON|SUBSCRIBES|ACCOUNT_OF*..4]- ... と書けることを口頭で確認
- 受講者への問い:「解約済みの顧客は除きたい」というCQ追加が来たらどこを変える?(SUBSCRIBESに status か期間を持たせる/契約を中間ノードに)
-->

## 命名規約と粒度の決め方

**命名規約(Text2CypherとLLM抽出の精度に直結)**

| 対象 | 規約 | 例 |
|---|---|---|
| ノードラベル | 単数形・PascalCase | `Person`, `Department` |
| リレーション | 動詞・UPPER_SNAKE、方向が読める | `WORKS_AT`, `DEPENDS_ON` |
| プロパティ | camelCase | `hiredAt`, `customerId` |

**粒度:ノードか、プロパティか(2章の判断を設計手順に)**

- CQで**そこを起点・経由に辿るか?** → 辿るならノード
- 例:「Python経験者は?」というCQがある → `Skill` は**ノード**
- CQに出ない属性(入社日など)はプロパティで十分

<!--
- 命名は美学ではなく精度の問題:LLMは名前の意味を手がかりにスキーマを使う。人間に読めない名前はLLMにも使えない
- リレーション名は「(from)が(to)を〜する」と声に出して自然かでチェック:Person WORKS_AT Company は自然、逆向きなら不自然と気づける
- 粒度の判断基準は一貫して「CQで辿るか」。汎用の正解はなく、CQが変われば答えも変わる、と相対化しておく
- 迷った例を受講者に投げる:「メールアドレスはノードか?」→ 通常はプロパティ。ただし不正検知で共有アドレスを辿るCQがあるならノード
-->

## グラフスキーマ → LLM抽出用JSONスキーマ

さっきのスキーマを structured output 用に変換(7章でそのまま使う):

```json
{
  "type": "object",
  "properties": {
    "entities": { "type": "array", "items": {
      "type": "object",
      "properties": {
        "id":    { "type": "string", "description": "正式名に正規化した名称" },
        "label": { "type": "string",
                   "enum": ["Incident","System","Service","Customer","Person"] }
      }, "required": ["id","label"] } },
    "relations": { "type": "array", "items": {
      "type": "object",
      "properties": {
        "type": { "type": "string",
                  "enum": ["AFFECTS","DEPENDS_ON","SUBSCRIBES","ACCOUNT_OF"] },
        "from": { "type": "string" }, "to": { "type": "string" }
      }, "required": ["type","from","to"] } }
  }, "required": ["entities","relations"]
}
```

<!--
- 「オントロジー=LLMを制約するスキーマ」の実物がこれ。ラベルと関係型がenumに落ちている点を指で示す
- 変換は機械的:ノードラベル→entities.label のenum、リレーション型→relations.type のenum。だから単一ソースから自動生成できる
- description フィールドはLLMへの指示文として機能する(「正式名に正規化」= 実用価値3の埋め込み)
- domain/range(AFFECTSの from は Incident のみ等)はJSONスキーマだけでは強制しきれない → 受け側のバリデーションコードで検査(次々スライド)
- このJSONを演習で各自が自分のドメインについて作る、と予告
-->

## 既存語彙の再利用:schema.org というカタログ

**車輪の再発明を防ぐ**:よくある概念は先人が定義済み

| 自作しがちな名前 | schema.org の語彙 |
|---|---|
| `Human` / `Member` / `Staff` | `Person` |
| `Company` / `Corp` | `Organization` |
| `worksAt` / `employedBy` | `worksFor` / `memberOf` |
| `Article` / `Doc` | `Article`, `Report`, `TechArticle` |

- 使い方:**命名と定義を借りる**(全部準拠する必要はない)
- ドメイン特化語彙の存在を知っておく:**FIBO**(金融)、**SNOMED CT**(医療)、**GoodRelations**(EC)
- 副次効果:LLMは公開語彙を**学習済み** → 抽出・生成の精度が上がりやすい

<!--
- schema.orgはGoogle等が検索用に策定した語彙集。「概念の命名辞典」として眺めるだけでも設計の質が上がる
- 「準拠」ではなく「参照」というスタンスを強調:社内KGでフル準拠は過剰。迷ったら見に行くカタログ
- 最後の副次効果は本章ならではの論点:LLMの事前学習にschema.orgが大量に含まれるため、独自語彙よりLLMが正しく扱いやすい
- FIBO/SNOMEDは「存在を知っている」レベルでOK。自分の業界に標準語彙があるか一度は検索してみるよう促す
- 6章でWikidataの語彙・外部IDと接続する話に繋がる、と予告
-->

## 制約の実務的な使いどころ

| 制約 | 例 | 実装先 |
|---|---|---|
| **domain / range**(関係の両端の型) | `WORKS_AT`: Person → Company のみ | 抽出後のバリデーションコード |
| **カーディナリティ** | 部署への所属は1つだけ(主務) | 投入時チェック / 名寄せの手がかり |
| **必須プロパティ** | Customer は customerId 必須 | JSONスキーマ `required` |
| **ID一意性** | sku, customerId | Neo4j 一意性制約(4章) |

- 目的は「哲学的正しさ」ではなく **LLM出力の検品**と**データ品質の防波堤**
- OWLの公理・推論器による整合性検査は本コースでは扱わない → **付録E**

<!--

## アンチパターン:オントロジー設計が失敗する形

| アンチパターン | 症状 | 処方箋 |
|---|---|---|
| **過度な抽象化** | `Thing→Agent→LegalEntity→…` 5階層の下に実データ | CQに出る概念だけ。階層は2〜3段まで |
| **哲学的正しさの追求** | 「役職はイベントか状態か」で会議が空転 | どちらでもCQが解ければ正解 |
| **使われない語彙の量産** | 関係型50種、使用実績3種 | 使う予定のないものは定義しない |
| **完璧主義で凍結** | 半年設計してデータゼロ | CQ10個分で作り始めて育てる |

> 判定基準はただ一つ:**そのスキーマでCQが解けるか、LLMが守れるか**

## LLMにオントロジードラフトを作らせる

```text
あなたはナレッジグラフのスキーマ設計者です。
以下のコンピテンシークエスチョンに答えられる最小のグラフスキーマを
設計してください。

# コンピテンシークエスチョン
1. ○○の障害の影響を受ける顧客と担当営業は?
2. Pythonのスキルを持つ社員は誰か? …

# 制約
- ノードラベル: 単数形PascalCase / リレーション: 動詞UPPER_SNAKE
- schema.org に対応する概念があれば同じ命名を使う
- 各CQがどのパターンで解けるか、Cypherのパターン例を添える
- ノードは8種以内、リレーションは10種以内

# 出力形式: ラベル一覧 / リレーション一覧(from,to,説明)/ CQ検証
```

**ワークフロー**:LLMがドラフト → **人間がCQでレビュー** → 確定 → JSONスキーマ生成

## 【演習】社内ナレッジのCQ→スキーマ+JSONスキーマ

題材:社内ナレッジ(**人・組織・スキル・ドキュメント**)

1. **CQを5個以上書く**(10分)
   - 実際に社内で聞かれる質問文で。例:「○○案件を手伝える有識者は?」

4. **ペアレビュー**(10分)

## 第5章まとめ

- オントロジー = **LLMを制約するスキーマ**(哲学ではなくエンジニアリング)
- 実用価値3点:**①抽出のスキーマ制約 ②Text2Cypherへの入力 ③用語正規化辞書**
- KGは**開世界**:知識は常に不完全、後から足せる — ただしID制約と単一ソース管理は死守
- 設計は**コンピテンシークエスチョン駆動**:答えたい質問→スキーマ→CQで検証
- グラフスキーマは機械的に**JSONスキーマ(enum制約)**へ変換できる
- schema.org をカタログとして再利用、過度な抽象化・哲学的正しさはアンチパターン
- LLMにドラフトを作らせてよい。**レビューとCQ検証は人間の仕事**
