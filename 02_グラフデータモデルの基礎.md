## グラフの構成要素:ノードとエッジ

```
        [購入]           [購入]
(佐藤) ──────→ (商品X) ←────── (鈴木)
   │                              │
   └────────[友人]────────────────┘
```

| 用語 | 意味 | ECサイトでの例 |
|---|---|---|
| **ノード**(頂点) | 「もの」「概念」 | 顧客、商品、注文 |
| **エッジ**(辺) | ノード間の関係 | 購入した、レビューした |
| **有向グラフ** | エッジに向きがある | 「佐藤 →購入→ 商品X」 |

- 本コースで扱うグラフは**有向グラフ**(向きに意味がある)
- ER図の「エンティティと関連」とほぼ同じ発想 — 実は既知の概念

<!--
- 「グラフ=棒グラフ・折れ線グラフ」ではなく数学のグラフ理論の意味だと最初に注意(初回は必ず誤解する人がいる)
- ER図との類似を強調:エンティティ≒ノード、リレーションシップ≒エッジ。ただしER図は設計図、グラフは実データ自体という違いは後で効いてくる
- 「友人」のような同種ノード間の関係はRDBだと自己結合になる、と軽く伏線を張る
- 無向グラフもあるが、実装上は有向で持ち「向きを無視して辿る」のが普通(4章で再説)
-->

## パス・次数・トラバーサル

```
(佐藤)→(注文1)→(商品X)←(注文2)←(鈴木)   ← 佐藤から鈴木への長さ4のパス
```

| 用語 | 意味 | クエリでの使われ方 |
|---|---|---|
| **パス** | エッジを辿った経路 | 「AとBはどう繋がっている?」 |
| **ホップ** | パスの1ステップ | 「3ホップ以内の関連を探す」 |
| **次数** | ノードに接続するエッジ数 | 「接続の多いハブを探す」 |
| **トラバーサル** | グラフを辿る操作 | グラフDBの基本動作 |

- 第1章の「マルチホップ質問」= **長いパスを辿る質問**だった
- RDBの JOIN 1回 ≒ グラフの 1ホップ — この対応が第3章の主役

<!--
- 第1章のデモを語彙で言い直す:「共通の取引先」=2ホップのパス発見、「影響範囲」=依存エッジのトラバーサル
- 「JOIN 1回=1ホップ」の対応は本コースで繰り返し使う中心的なアナロジー。ここで刷り込む
- 次数の話は11章の中心性分析への伏線。「フォロワー数=入次数」と言うとSNS経験から腹落ちしやすい
- 質問例:「みなさんのDBで一番JOINが深いクエリは何段ですか?」→ 3章の導入になる
-->

## プロパティグラフモデル — 本コースの標準

**ノードとエッジの両方が「ラベル/型」と「プロパティ」を持てる**グラフモデル

| 構成要素 | 持てるもの | 例 |
|---|---|---|
| ノード | **ラベル**(種類)+ プロパティ | `:Customer {name:"佐藤", since:2021}` |
| リレーション | **型** + 方向 + プロパティ | `-[:REVIEWED {rating:5, date:"2026-05-01"}]->` |

```
(:Customer {name:"佐藤"}) -[:REVIEWED {rating:5}]-> (:Product {name:"キーボードK1"})
```

- Neo4j、Amazon Neptune、Memgraph などが採用する事実上の標準
- **リレーションにプロパティを持てる**のが最大の特徴(RDFとの違い)

<!--
- 「プロパティグラフ」という名前の由来=プロパティを持てるグラフ、とそのまま覚えてもらう
- ラベルとプロパティの区別を丁寧に:ラベルは「種類の分類」(テーブル名に相当)、プロパティは「個体の属性」(列に相当)
- リレーションのプロパティの実用例を口頭で追加:「取引額」「いつからの友人か」「レビューの点数」— 関係自体がデータを持つ
- この記法 `(:Label {prop})-[:TYPE]->()` はCypher(4章)そのもの。今から目に慣らしておくと4章が楽
-->

---

## 実例:ECサイトのプロパティグラフ

```
(:Customer {id:"c01", name:"佐藤", region:"関東"})
     │
  [:PLACED {at:"2026-06-01T10:23"}]
     ▼
(:Order {id:"o1001", total:12800, status:"配送済"})
     │
  [:CONTAINS {qty:2, unit_price:3400}]
     ▼
(:Product {id:"p42", name:"キーボードK1", price:3400})
     │
  [:BELONGS_TO]
     ▼
(:Category {name:"PC周辺機器"})
```

- ラベル4種、リレーション型3種の小さなグラフ
- この例を**本章の軸**として、RDBとの対応・設計判断を見ていく

<!--
- 本章のランニングエグザンプル。演習でもこのドメインを拡張するので構造を頭に入れてもらう
- 各要素を指差し確認:「PLACEDのプロパティは注文日時。なぜOrderノードではなくエッジに?→どちらでも成立する。この判断を後半でやります」と設計問題への興味を引く
- CONTAINSのqty・unit_priceに注目させる:「これはRDBだと何テーブルの何列ですか?」→ order_items。次の対応表への布石
- ラベルは単数形・パスカルケース、リレーション型は動詞・大文字スネークが慣例(5章の命名規約で再説)
-->

## RDB → プロパティグラフ 対応表

| RDB | プロパティグラフ | ECサイトの例 |
|---|---|---|
| テーブル | **ノードラベル** | `customers` → `:Customer` |
| 行 | **ノード** | 佐藤さんの行 → 佐藤ノード |
| 列 | **プロパティ** | `name`, `region` |
| 主キー | 一意制約付きプロパティ | `id:"c01"` |
| 外部キー | **リレーション** | `orders.customer_id` → `[:PLACED]` |
| 中間テーブル | **リレーション+プロパティ** | `order_items` → `[:CONTAINS {qty}]` |
| JOIN | **トラバーサル**(パターンマッチ) | 3章・4章の主題 |

> この1枚が本章の中心。**RDBスキーマが読めればグラフはモデリングできる**

<!--
- 本章で最重要のスライド。時間をかけ、1行ずつECの例と往復しながら進める
- 外部キーの行が概念の転換点:RDBでは「値の参照」だった関係が、グラフでは「実体を持つエッジ」に昇格する。ここが腹落ちすれば勝ち
- 中間テーブル→リレーション+プロパティは次のスライドで図解する、と予告
- 注意も一言:「機械的な変換はスタート地点であってゴールではない。クエリ要件で崩す判断を後半でやる」
- 対応表は付録Cのチートシートにも収録していると案内
-->

## 外部キーはリレーションになる

<div class="columns">

<div>

**RDB(参照は値)**

```sql
CREATE TABLE orders (
  id          TEXT PRIMARY KEY,
  customer_id TEXT REFERENCES
              customers(id),
  total       INT
);
-- 関係を見るには JOIN が必要
SELECT c.name, o.total
FROM orders o
JOIN customers c
  ON c.id = o.customer_id;
```

</div>
<div>

**グラフ(参照は実体)**

```
(:Customer {id:"c01"})
   -[:PLACED]->
(:Order {id:"o1001"})
```

- 関係が**最初から繋がって格納**される
- クエリは繋がりを**辿るだけ**
- 「どう繋がるか」をクエリ時に計算
  (JOIN)するか、格納時に持つか
  — 思想の違い

</div>
</div>

<!--
- 左右対比で「同じ情報の持ち方の違い」を見せる。情報量は同じ、構造の表現が違う
- 「JOINは毎回関係を計算で復元している」という言い方が3章のindex-free adjacencyへの最短の伏線になる
- RDBが劣っているという話ではない:JOINの柔軟性(どの列でも突き合わせられる)はRDBの強み。グラフは「事前に宣言した関係」しか辿れない
- 受講者への問いかけ:「外部キー参照を毎秒何万回もJOINで復元しているシステム、心当たりは?」
-->

## 中間テーブルはリレーション+プロパティになる

**RDBの多対多**:`order_items(order_id, product_id, qty, unit_price)`

```
RDB:   orders ──< order_items >── products     (テーブル3つ+JOIN 2回)

グラフ: (:Order)-[:CONTAINS {qty:2, unit_price:3400}]->(:Product)
                                                        (エッジ1本)
```

- 中間テーブルの**外部キー2本 → エッジの両端**
- 中間テーブルの**残りの列 → エッジのプロパティ**
- 多対多のJOIN 2回が、1ホップのトラバーサルに畳み込まれる

⚠ ただし中間テーブルが「第3の実体」なら**ノードに昇格**(次の判断基準へ)

<!--
- 中間テーブルの2種類を見分けるのがポイント:純粋な関連(order_items)はエッジ、実体を持つ関連はノード
- 「第3の実体」の判定例を口頭で:雇用(employments)は入社日・役職・部署と繋がる→Employmentノードにしたくなる。これが4章の中間ノードパターン
- order_itemsも「行ごとに返品状態を管理したい」等の要件が付くとノード化が候補になる。要件次第で答えが変わる最初の例
- 最後の⚠が次のスライドへのブリッジ。「では昇格の判断基準は?」
-->

## 判断基準:ノードか、プロパティか、リレーションか

| 選択 | 目安(YESが多ければ) |
|---|---|
| **ノード**にする | ・それ自体を起点/終点に検索したい<br>・複数の関係の交差点になる<br>・固有のIDで同一性を管理したい |
| **プロパティ**にする | ・単一の値で、そこから他を辿らない<br>・フィルタ・表示にしか使わない |
| **リレーション**にする | ・2つのエンティティ間の事実である<br>・「誰が/何に/どうした」と読める |

> 決め手は構造の美しさではなく **「どんな質問(クエリ)に答えたいか」**

<!--
- 本章2番目の重要スライド。「正解はクエリ要件が決める」を合言葉として導入する(5章のコンピテンシークエスチョンの原型)
- RDBの正規化との対比が効く:正規化は更新異常の排除が基準、グラフモデリングは問い合わせパターンが基準。基準が違う
- 「交差点になる」の直感的な言い換え:「そこで曲がりたいならノード、通り過ぎるだけならプロパティ」
- 迷ったらプロパティで始めてよい:ノード昇格のリファクタリングは可能(コストの話は口頭で軽く)
- 次の2枚で実際に例題を解く、と予告
-->

## 例題1:「商品カテゴリ」はどうする?

`products.category = "PC周辺機器"` — 列(プロパティ)のままでよい?

| 答えたい質問 | 適した設計 |
|---|---|
| 「この商品のカテゴリを表示したい」だけ | プロパティで十分 `{category:"PC周辺機器"}` |
| 「同じカテゴリの商品を推薦したい」 | **ノード** `(:Product)-[:BELONGS_TO]->(:Category)` |
| 「カテゴリ階層(周辺機器⊂PC用品)を辿りたい」 | **ノード+自己参照** `(:Category)-[:CHILD_OF]->(:Category)` |

- 同じデータでも**クエリ要件が変われば設計が変わる**
- カテゴリを「交差点」として使う(商品同士を繋ぐ)ならノード

<!--
- 判断基準の初適用。上から順に要件を追加していき、設計が段階的に変わる様子を見せる
- 2行目の推薦クエリを図で補足:商品A→カテゴリ→商品B と辿れる。プロパティだと「同じ文字列を持つ行を探す」照合になり、グラフの旨味がない
- 3行目はRDBだと自己参照FK+再帰CTEになる例。「3章でこのSQLの苦しさを見ます」と伏線
- 受講者に問う:「タグ(多対多)ならどうしますか?」→ ノード。カテゴリと同じ理屈で考えられたら合格
-->

---

## 例題2:「配送先住所」はどうする?

`orders.shipping_address = "東京都港区..."` — どう持つ?

| 答えたい質問 | 適した設計 |
|---|---|
| 「注文の配送先を表示したい」だけ | **プロパティ**で十分 |
| 「同じ住所に届く注文をまとめたい」(同一世帯検出) | **ノード** `(:Order)-[:SHIP_TO]->(:Address)` |
| 「地域ごとの購買傾向を分析したい」 | ノード+地域階層 `(:Address)-[:IN]->(:Region)` |

- **不正検知の定番**:別人を装う複数アカウントが同一 `Address` ノードを共有
  → 「関係のパターン」が見える(1章の不正検知ユースケースの正体)

<!--
- 例題1と同じ思考プロセスをもう一周して定着させる。今度は受講者に先に答えを予想させてから表を出す
- 不正検知の例は「住所をノードにした瞬間、住所がアカウント同士を繋ぐハブになる」ことを図解すると強い。プロパティのままでは絶対に見えない構造
- 名寄せ問題(表記ゆれで同じ住所が別ノードになる)にここで軽く触れ、8章のエンティティ解決で本格的に扱うと予告
- まとめの一言:「ノード化とは、その概念を検索と分析の一級市民に昇格させること」
-->

## もう一つのモデル:RDFトリプル(概要のみ)

**主語–述語–目的語**の3つ組(トリプル)だけで全てを表す W3C 標準モデル

```turtle
<佐藤>  <購入した>  <キーボードK1> .
<キーボードK1>  <価格>  3400 .
```

| 観点 | プロパティグラフ | RDF |
|---|---|---|
| 標準化 | 事実上の標準(GQLでISO標準化) | W3C標準(IRI・語彙も標準化) |
| エッジの属性 | プロパティで直接持てる | 素直には持てない(RDF-star等) |
| 主な世界 | アプリのグラフDB(Neo4j等) | 公開KG(Wikidata)、相互運用 |

**本コースはプロパティグラフが主軸**。RDFは第6章(外部知識の取り込み)で実践

<!--
- ここは深入りしないと明言する。「両者の位置づけを知り、6章で再会したとき混乱しない」ことだけがゴール
- 「全部を3つ組に分解する」という思想だけ伝える:プロパティも `<商品> <価格> 3400` というトリプルになる。徹底的に均質なモデル
- どちらが優れているかではなく生息域が違う:社内システム=プロパティグラフ、Web規模の公開データ=RDF、と覚えると実務で迷わない
- 1章の用語レイヤ図を再掲してもよい:「KGの表現形式が2つある」という位置づけの確認
-->

## LLMとの親和性:抽出出力はそのままグラフ

LLMに文書からの知識抽出を頼むと、自然に**トリプル形式**が出てくる:

```json
{"subject": "佐藤",   "relation": "所属", "object": "営業部"}
{"subject": "営業部", "relation": "担当", "object": "X社"}
```

↓ そのままプロパティグラフに

```
(:Person {name:"佐藤"})-[:所属]->(:Org {name:"営業部"})-[:担当]->(:Company {name:"X社"})
```

- 「エンティティと関係」という抽出の単位が、ノードとエッジに**1対1対応**
- 関係の種類が後から増えても、テーブル追加・ALTER不要 → 抽出結果を受け止めやすい
- これが第7章「LLMによるKG自動構築」の土台になる

<!--
- 本章と後半章を繋ぐ伏線スライド。「なぜコースの標準がグラフモデルなのか」の答え合わせ
- RDBで同じことをやる想像をしてもらう:抽出のたびに未知の関係が出てきたら中間テーブルを何個作る?スキーマ確定できるのか?→ グラフの柔軟性が効く
- ただし柔軟=無法ではない。「許可する型を制約しないとLLMは好き放題な関係を作る」→ 5章のオントロジーで制御する、と両面を予告
- ここは深追いせず1分程度で。「後で戻ってくる場所」という位置づけ
-->

## グラフアルゴリズムの入り口

構造そのものから価値を引き出す計算群(詳細と実装は第11章)

| アルゴリズム | 答える問い | ECサイトでの例 |
|---|---|---|
| **最短経路** | AからBへ最少ホップで? | 顧客同士の繋がりの近さ |
| **中心性** | どのノードが「重要」か? | ハブ商品、影響力のある顧客 |
| **コミュニティ検出** | 自然なグループ分けは? | 購買傾向のクラスタ発見 |

- クエリ(特定の答えを取る)とアルゴリズム(全体構造を分析する)は別物
- 「グラフにしておくと後からこの分析が全部使える」のも Graph DB 選定理由の一つ

## 第2章まとめ

- グラフの語彙:**ノード・エッジ・パス・トラバーサル**(JOIN 1回 ≒ 1ホップ)
- 本コースの標準は**プロパティグラフ**:ラベル付きノード+型付きリレーション+プロパティ
- RDBとの対応:**テーブル→ラベル、外部キー→リレーション、中間テーブル→リレーション+プロパティ**
- ノード/プロパティ/リレーションの判断は**「答えたい質問」が決める**
- RDFトリプルはもう一つの表現形式(第6章で実践)。LLM抽出との親和性は第7章の土台

**次章**:RDBMSでグラフを扱う限界と Graph DB を使うべき要件
— 「今のRDBのままで粘るか、移行するか」を判断できるようになる
