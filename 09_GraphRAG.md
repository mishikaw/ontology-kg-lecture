## ベクトルRAGの限界:構造的な整理

| 限界 | 原因 | 失敗する質問 |
|---|---|---|
| **チャンク分断** | 分割時に文書間・文書内の**関係が切断**される | 「AとBの共通取引先は?」 |
| **マルチホップ不能** | 類似度検索は1回の「意味の近さ」しか見ない | 「障害の影響範囲は?」(A→B→Cと辿る) |
| **集約不能** | 上位k件しか取れない=**全件走査できない** | 「製造業の顧客は何社?」 |

- どれも**embeddingの精度向上では解決しない**:情報の持ち方の問題
- RDB経験者向けの整理:
  - ベクトル検索 ≒ 全文検索インデックス(**WHERE句しかないSQL**)
  - 欲しいのは **JOIN(関係の追跡)と GROUP BY(集約)** に相当する能力
  - KG+グラフクエリがそれを提供する

<!--
- 1章では現象として見せた失敗を、ここでは3つのメカニズムに分類して名前を付ける。「説明できる」状態にするのが狙い
- 「WHERE句しかないSQL」の比喩は本章で一番持ち帰ってほしい表現。類似検索は選択(selection)はできるが結合(join)と集約(aggregation)ができない
- 「チャンクを大きくすれば?」という質問が必ず出る:文書をまたぐ関係は救えない、コンテキスト浪費で精度も落ちる、と先回りして潰す
- 公平性の担保:単純な事実検索ならベクトルRAGで十分・高速・安価。「弱点の整理」であって「全否定」ではない(ルーティングの伏線)
-->

## GraphRAGの共通骨格と4類型

**共通骨格:質問 → [KGから関連構造を取得] → コンテキスト化 → LLMが回答**

違いは「**KGからどう取得するか**」— 4つの類型:

| 類型 | 取得方法 | 一言で |
|---|---|---|
| **① Text2Cypher** | 質問をクエリに翻訳して実行 | LLMがSQLならぬCypherを書く |
| **② グラフ探索型** | エンティティを起点に近傍を展開 | 起点から網を広げる |
| **③ ハイブリッド** | ベクトル検索で起点発見→グラフ展開 | ②の起点探しをベクトルで解決 |
| **④ コミュニティ要約型** | 事前に階層要約を構築して参照 | 全体俯瞰の質問に答える |

- ①②③はクエリ時に取得、④は**インデックス構築時**に勝負が決まる
- これから1類型ずつ:仕組み・得意な質問・弱点

<!--
- 「GraphRAG=1つの手法」という誤解を最初に解く。retrieverの設計が違う4家族であり、どれを選ぶかが設計判断
- ベクトルRAGとの対比で言えば「retrieval部分だけが違い、生成部分は同じ」。RAGの経験がそのまま活きる
- ①〜③がオンライン(クエリ時)、④がオフライン(構築時)という軸は覚えやすい整理なので板書する
- 後で比較表(このあと必須スライド)に戻ってくるので、細部を覚えようとしなくてよいと安心させる
-->

## 類型①:Text2Cypher — 自然言語をクエリに翻訳

```
質問「製造業の顧客は何社?」
  ↓ LLM(プロンプトにグラフスキーマを同梱)
MATCH (c:Company {industry: "製造業"})-[:顧客]->(:自社)
RETURN count(c)
  ↓ Neo4jで実行 → 「12」→ LLMが文章化
```

| 得意な質問 | 弱点 |
|---|---|
| **集約・カウント**(何社?最多は?) | クエリ**誤生成**(構文/スキーマ違反/意味の取り違え) |
| 条件検索・ソート(2024年以降の契約は?) | 曖昧な質問に弱い(条件をクエリにできない) |
| スキーマに沿った定型的な質問 | 表記ゆれ:「トヨタ」で聞かれてもノードは「トヨタ自動車」 |

- 1章デモの「製造業の顧客は何社?」は**この類型**が答えていた
- SQL経験者に馴染むText2SQLのグラフ版 — 成否は**スキーマの伝え方**で決まる

<!--
- 種明かし第1弾:集約質問はText2Cypherだった、とデモと明示的に紐付ける
- Text2SQLをやったことがある受講者に挙手してもらう。「あれと同じ構造。成功も失敗も同じパターン」
- 表記ゆれの弱点は8章の名寄せと繋がる:aliasesプロパティやフルテキストインデックスを検索してからクエリに埋める対策がある

## Text2Cypher:スキーマの与え方と誤生成対策

```text
あなたはCypherの専門家。以下のスキーマだけを使いクエリを生成せよ。
# ノード: Company(name, industry), Person(name), Service(name)
# 関係: (Person)-[:担当]->(Company), (Company)-[:契約]->(Service)
# 例: Q:「A社の担当者は?」
#     A: MATCH (p:Person)-[:担当]->(c:Company {name:"A社"}) RETURN p.name
制約: 読み取り専用(MATCH/RETURNのみ)。存在しないラベル・関係を使わない。
質問: {question}
```

| 対策 | 内容 |
|---|---|
| スキーマ同梱 | ラベル・関係型・プロパティ+**few-shot例**(5章の成果物) |
| 実行前検証 | `EXPLAIN`で構文チェック/スキーマ要素の存在確認 |
| 権限分離 | **読み取り専用ユーザー**で実行(DELETE事故の遮断) |
| 失敗時リトライ | エラーメッセージを添えて再生成(2〜3回で打ち切り) |
| 結果ゼロ時 | 「該当なし」と正直に返す(**推測で埋めない**) |

<!--
- 5章の「オントロジーはText2Cypherへの入力」という伏線をここで回収。スキーマ記述の質=クエリ生成精度
- few-shot例は自社の頻出質問パターンから作る。汎用例より自社スキーマに即した例が効く
- 権限分離はRDBの常識と同じ:「アプリ用DBユーザーにDROP権限を与えない」。LLM生成クエリなら尚更、と言えば全員頷く
- エラーを添えたリトライで成功率が大きく上がるのが実務の実感。ただし無限ループ防止の上限は必須(10章のエージェント信頼性設計の先取り)
- 「該当なし」を正直に返すのはハルシネーション抑制の要。1章デモの「存在しない関係は見つからないと答える」の裏側
-->

## 類型②:グラフ探索型 — エンティティ起点で近傍展開

```
質問「X社との関係を教えて」
  ↓ ①質問からエンティティ抽出(LLM): "X社"
  ↓ ②グラフ上のノードに対応付け(8章の名寄せ技術)
  ↓ ③近傍サブグラフを取得(1〜2ホップ)
MATCH path = (c:Company {name:"X社"})-[*1..2]-(x)
RETURN path LIMIT 50
  ↓ ④サブグラフをテキスト化してLLMに渡す → 回答
```

| 得意な質問 | 弱点 |
|---|---|
| エンティティ中心の質問(「X社について」) | **起点が見つからないと全滅**(表記ゆれ・曖昧な質問) |
| 関係の説明(「XとYの繋がりは?」) | ホップ数を広げると**爆発**(コンテキスト溢れ) |
| Cypherにしにくい「ざっくり質問」 | 集約質問は苦手(近傍しか見ないため) |

<!--
- Text2Cypherが「クエリで正確に切り出す」のに対し、こちらは「起点の周りをまるごと取ってLLMに読解させる」。厳密さと柔軟さのトレードオフ
- ②のエンティティ対応付けで8章の名寄せ技術(正規化・embedding照合)がそのまま再利用される。8章は構築のためだけの技術ではなかった
- ホップ爆発は具体的な数字で:平均次数20なら2ホップで400ノード。LIMITと関係型の絞り込みが必須
- サブグラフのテキスト化は「(X社)-[契約]->(サービスP)」のようなトリプル列挙が単純で効果的、と実装イメージを添える
- 弱点「起点が見つからない」の解決策こそが次のハイブリッド、と流れを作る
-->

## 類型③:ハイブリッド — ベクトルで起点発見→グラフ展開

**実務の第一選択。**4章のベクトルインデックスがここで効く

```
質問「バッテリー技術で提携している会社は?」
  ↓ ①質問をembedding化 → ノード/チャンクをベクトル検索
     (「バッテリー技術」に意味的に近いノードがヒット)
  ↓ ②ヒットしたノードを起点にグラフ展開(retrieval query)
  ↓ ③テキスト根拠+グラフ構造の両方をコンテキストに → 回答
```

- **両者の弱点を相殺**:
  - ベクトル検索が「曖昧な言葉 → 起点ノード」を解決(類型②の弱点)
  - グラフ展開が「関係の追跡」を解決(ベクトルRAGの弱点)
- 1章デモの「共通の取引先」「障害の影響範囲」は**この類型**が答えていた

<!--
- 種明かし第2弾:1章デモの主役はハイブリッドだった。マルチホップ系の質問はこれで答えていた
- 「実務の第一選択」の理由を明確に:既存のベクトルRAGからの増築で作れる/曖昧な自然言語質問に強い/Text2Cypherより誤動作が穏やか(クエリ誤生成ではなく検索精度の低下として現れる)
- 4章でベクトルインデックスを学んだとき「9章で使う」と言った伏線の回収。ノードにembeddingを持たせた意味がここで分かる
- チャンクノードをKGに含める設計(チャンク-[:MENTIONS]->エンティティ)を口頭で補足:テキストと構造を1つのグラフで往復できるのがNeo4j流
- 次の2枚で実際のコードを見せる、と予告
-->

## ハイブリッド実装(1):Neo4j GraphRAG パッケージ

```python
from neo4j_graphrag.retrievers import VectorCypherRetriever
from neo4j_graphrag.generation import GraphRAG

retriever = VectorCypherRetriever(
    driver, index_name="chunk_embeddings",
    retrieval_query="""
    // ベクトル検索でヒットした node(チャンク)から:
    MATCH (node)-[:MENTIONS]->(e)-[r]-(nb)     // 言及エンティティ→近傍へ展開
    RETURN node.text AS chunk,
           e.name + ' -[' + type(r) + ']- ' + nb.name AS graph_context
    """,
)
rag = GraphRAG(retriever=retriever, llm=llm)
print(rag.search("AさんとBさんの共通の取引先は?").answer)
```

- `retrieval_query` が心臓部:**ベクトル検索の続きをCypherで書く**
- ここに8章のプロビナンス(`source_text`)を返せば**根拠付き回答**になる

<!--
- コードの読みどころは1点だけ:retrieval_queryの中のnode変数。「ベクトル検索の結果がCypherの変数として渡り、そこから自由に展開できる」— これがハイブリッドの実装のすべて
- 7章でSimpleKGPipeline(構築)を使ったのと同じパッケージの活用側。構築と活用が同一エコシステムで繋がる
- retrieval_queryを要件に合わせて書き換えるのが実務の腕の見せどころ:担当者を辿る、依存チェーンを辿る、など。4章のCypher力がそのまま活きる
- 「根拠パスはどこから来た?」(冒頭の疑問3)の最終回答:retrieval_queryでsource_textとパスを返し、回答に添えていた
-->

## 類型④:コミュニティ要約型(Microsoft GraphRAG)

**「このデータ全体の主要なテーマは?」— 局所検索では原理的に答えられない質問へ**

```
[構築時(オフライン)]
KG → コミュニティ検出(Leiden法)でクラスタ分割
   → 各コミュニティをLLMで要約 → 階層的に上位要約を生成
[クエリ時]
全体俯瞰質問 → 関連コミュニティ要約を収集 → 部分回答を統合(map-reduce)
```

| 得意な質問 | 弱点 |
|---|---|
| **全体俯瞰**(主要テーマは?全体動向は?) | **インデックス構築コストが莫大**(全グラフをLLM要約) |
| 「データセット全体を読まないと答えられない」質問 | 更新のたびに要約の再構築が必要 |
| 網羅的なレポート生成 | 個別事実の質問には**過剰装備** |

<!--
- 発想の転換を強調:①〜③は「クエリ時にどう探すか」の工夫、④は「構築時に答えを仕込んでおく」。DWHの集計テーブル事前作成に似ている、とRDB対比
- コミュニティ検出は2章で予告したグラフアルゴリズム(11章で本格活用)の実戦投入例

## 評価:GraphRAGの良し悪しをどう測るか

**RAG評価の基本指標(ベクトルでもグラフでも共通):**

| 指標 | 問い | 悪いときの症状 |
|---|---|---|
| **Faithfulness** | 回答は取得コンテキストに忠実か | 幻覚(根拠なき生成) |
| **Answer Relevancy** | 回答は質問に答えているか | 的外れ・冗長な回答 |
| **Context Precision** | 取得した情報に無駄がないか | ノイズ混入→精度低下・コスト増 |
| **Context Recall** | 必要な情報を取得できたか | 「わかりません」・不完全な回答 |

- 前半2つ=**生成の質**、後半2つ=**検索(retrieval)の質**
- GraphRAGの改善はほぼ**retrieval側**(retrieval query・ルーティング)で行う
- 測り方は8章と同じ思想:**評価用質問セット(ゴールド)**+LLM-as-a-judge

<!--
- 8章の評価は「KG構築の品質(トリプルのP/R)」、本章は「QAシステムの品質」。レイヤが違うことを最初に整理する
- 「生成側と検索側の切り分け」が診断の第一歩:faithfulnessが低い→プロンプト/モデルの問題、context recallが低い→retrieval設計の問題。障害切り分けと同じ思考
- これらの指標の多くはLLM-as-a-judgeで自動計算される=8章で学んだ限界(judge自身の誤り、校正の必要性)がそのまま適用される、と接続
- 評価用質問セットは1章演習で作った「差が出る質問」が原型として使える。コース全体が繋がっていることを実感させる
-->

## ベクトルRAGとのA/B比較方法

**「GraphRAGにして良くなったのか?」に数字で答える**

| 手順 | 内容 |
|---|---|
| 1. 質問セット設計 | **質問タイプ別に**用意:単純検索/マルチホップ/集約 各10問など |
| 2. 同一条件 | 同じ文書・同じLLM・同じ評価指標で両システムを実行 |
| 3. タイプ別に集計 | **全体平均で潰さない** — タイプごとに差を見る |
| 4. コストも並記 | 精度だけでなく応答時間・API費用も比較 |

結果イメージ(だから使い分ける):

| 質問タイプ | ベクトルRAG | GraphRAG(ハイブリッド) |
|---|---|---|
| 単純検索 | **0.91** | 0.89(同等・やや高コスト) |
| マルチホップ | 0.42 | **0.86** |
| 集約 | 0.30 | **0.83**(Text2Cypher併用) |

<!--
- 「全体平均で潰さない」が最重要ポイント:平均だと差が薄まり「大差ない」と誤結論になる。タイプ別に見ると景色が一変する
- この表が示す実務的な結論:単純検索はベクトルで十分(むしろ速くて安い)。GraphRAGの価値は特定タイプに集中している → 次のルーティングへの完璧な導入
- 数字はイメージであり実際は データ・実装依存。ただし「単純検索は互角、構造質問で圧勝」というパターンは典型的
- 社内提案での使い方:この表がそのままROI説明資料になる(11章)。「どのタイプの質問が業務にどれだけあるか」を掛け算すれば効果額が出る
- 演習でこの表を自分のKGで作ることを予告

## 質問タイプ別ルーティング:全部GraphRAGにしない

```python
ROUTER_PROMPT = """質問を分類せよ:
- simple:  単一の事実・文書検索       → ベクトルRAG
- multihop: 関係を辿る・影響範囲      → ハイブリッドGraphRAG
- aggregate: 件数・ランキング・集計   → Text2Cypher
- global:  全体傾向・テーマ           → コミュニティ要約(あれば)
質問: {question}  分類のみJSONで返せ: {"route": "..."}"""
```

- ルータは**小型モデルで十分**(分類タスク。8章の段階戦略)
- 効能:精度(得意な経路に流す)+**コスト・レイテンシ**(重い経路を温存)
- フォールバック設計:ルート先が空振り → 別ルートで再試行 → 正直に「不明」
- A/B比較の結果が**ルーティング表の根拠**になる(前スライドの表がそのまま設計書)

<!--
- 本章の実務的な結論:「ベクトル vs グラフ」の二択ではなく「質問を適材適所に振り分ける」が正解。1章から張ってきた「使い分け」の伏線を回収
- ルータ自体の精度も測定対象(分類の混同行列)。ルータが誤ると全部が狂うので、評価質問セットにルート正解ラベルも付けておくと良い
- 「わからない質問はどうする?」→ デフォルトルートを決めておく(通常はハイブリッド)。過剰に細かい分類より4分類程度の粗さが運用しやすい
- 10章への予告:ルーティングを静的な分類ではなくエージェントの動的な判断にする、という発展形がある
-->

## レイテンシとコスト設計

| 経路 | LLM呼び出し | 典型レイテンシ | 相対コスト |
|---|---|---|---|
| ベクトルRAG | 1回(生成) | 〜2秒 | 低 |
| ハイブリッド | 1回(生成) | 〜3秒(+グラフ探索) | 低〜中 |
| Text2Cypher | **2回**(生成×2) | 〜5秒 | 中 |
| グラフ探索型 | 2回(抽出+生成) | 〜5秒 | 中 |
| コミュニティ要約 | 多数(map-reduce) | 数十秒 | 高+**構築費** |

削減の定石(8章の道具がそのまま使える):

- **プロンプトキャッシュ**:スキーマ・few-shot部分は毎回同じ
- **セマンティックキャッシュ**:類似質問の回答を再利用(FAQは同じ質問が来る)
- グラフ探索の`LIMIT`・ホップ上限 → コンテキスト量=入力トークンの直接削減
- ルーティングそのものが最大のコスト対策(安い経路を第一候補に)

<!--
- 「Text2CypherはなぜLLM2回?」:クエリ生成で1回、結果の文章化で1回。呼び出し回数の構造で各経路のレイテンシが読める
- グラフ探索(Cypher実行)自体は通常ミリ秒〜数十ミリ秒で、支配項は常にLLM呼び出し。「遅い=グラフDBのせい」と誤診しない
- コンテキスト量の削減は品質にも効く:無駄な近傍ノードはノイズ(context precision低下)でもある。コストと品質が同じ方向を向く珍しい改善
- セマンティックキャッシュの注意:KGが更新されたら無効化が必要(8章の鮮度管理と接続)
- ユーザー体験の工夫も一言:ストリーミング表示や「グラフを探索中…」表示で体感レイテンシは大きく改善する
-->

## 第9章まとめ

- ベクトルRAGの構造的限界=**チャンク分断・マルチホップ不能・集約不能**
  - 「WHERE句しかないSQL」— JOINとGROUP BYに相当する能力をKGが足す
- **GraphRAG 4類型**:Text2Cypher / グラフ探索 / ハイブリッド / コミュニティ要約
  - 迷ったら**ハイブリッド**から。集約が増えたらText2Cypher併設
- 実装の心臓部は `retrieval_query`:**ベクトル検索の続きをCypherで書く**
- 評価は**質問タイプ別**に:Ragasで自動化し、全体平均で潰さない
- **ルーティング**で適材適所:全部GraphRAGにしない。それが精度とコストの両立
