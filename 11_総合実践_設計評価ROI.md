### 第11章 総合実践:ソリューション設計・評価・ROI

## 本章のゴール

- 要件→DB選定→スキーマ→構築→活用→運用の**一連の意思決定**を、各章の成果物を使って行える
- **KGを作らない判断**(ベクトルRAG/RDBMSで十分な要件)ができる
- 効果を**数字で示せる**:精度改善・時間短縮・知識の資産化 → ROI
- 運用・スケール・失敗パターンを踏まえた**現実的な提案書**を書ける

> 最終成果物:
> **技術選定理由・評価結果・ROI試算を含む提案書としてプレゼンする**

<!--
- 「一連の意思決定」が本章のキーワード。個々の技術は既習。順番に正しく判断を積むことが今日のスキル
- 「作らない判断」をゴールに明記しているのはこのコースの特徴。技術コースの最終章で「使わない話」をするのは、それが実務の信頼に直結するから
- ROIの話は苦手意識を持つエンジニアが多い。「数式は掛け算と引き算だけ。怖くない」と先に言っておく
- プレゼンがあることを冒頭で再度予告し、前半の講義を「使うつもり」で聞かせる
-->

## ケーススタディ:社内ナレッジグラフを作る

**題材**:中堅SIer(社員800名)の社内ナレッジ活用

| 現状の課題 | 具体例 |
|---|---|
| 属人化 | 「その件はSさんしか知らない」— Sさんは来月退職 |
| 調査に時間 | 「PostgreSQLの移行経験者は?」→ 各部に聞いて回って2日 |
| 知見が散在 | 議事録・日報・PJ完了報告が別システムに死蔵 |

**要望**:「誰が何を知っているか」「過去の類似案件」「障害対応の知見」に答えたい。
ゆくゆくは調査エージェントに任せたい。

→ この要件を、**各章の成果物で順に設計していく**

<!--
- 5章の演習(人・組織・スキル・ドキュメントのスキーマ)と同じドメイン。演習成果物をそのまま持ち込めるよう意図的に揃えている
- 「Sさん退職」のような固有の痛みを入れると要件がリアルになる。受講者に「自社ならどの課題が一番痛い?」と聞いて自分事化させる
- 「ゆくゆくはエージェント」という段階的な要望も現実的なポイント:最初から全部作らない、が後のスモールスタートに繋がる
- ここから4枚で要件→選定→設計→構築→活用を一気に追体験する。「初見の内容はゼロ。全部やったことの再配置」と安心させる
-->

## 設計フロー:各章の成果物を繋ぐ

```
要件定義          DB選定           スキーマ設計       構築              活用
CQを書き出す ──→ 3章チェック  ──→ 5章の手順で   ──→ 7章:LLM抽出  ──→ 9章:GraphRAG
「何の質問に      リストで          CQ→スキーマ       8章:名寄せ・     10章:エージェント
  答えるKGか」    要否を判定        +抽出用JSON       品質・コスト
        └────────────── 評価と運用:8章(品質・コスト)+ 9章(RAG評価)+ 本章(ROI・運用)──┘
```

| 工程 | 使う成果物 | 本ケースでの答え |
|---|---|---|
| 要件定義 | コンピテンシークエスチョン(5章) | 「X技術の有識者は?」など10問 |
| DB選定 | チェックリスト(3章) | → 次スライドで判定 |
| 設計〜活用 | 5章スキーマ → 7-8章パイプライン → 9-10章 | 以降のスライドで |

<!--
- コース全体が1本のフローに繋がる「地図」のスライド。ここで各章の位置づけがカチッとはまる瞬間を作りたい
- 順序が大事:「DBを選んでからスキーマ」ではなく「CQ(=何に答えたいか)が先、道具は後」。技術起点で始めるプロジェクトが失敗する理由はこの順序の逆転
- 評価と運用がフローの下に「常時走る帯」として描いてあることを指摘:最後に付け足すものではない
- 「みなさんの5章演習のCQとスキーマ、8章の品質レポート、9章の比較レポートが今日の提案書の素材になる」と成果物の再利用を促す
-->

## Step 1:DB選定 — 3章チェックリストを当てる

| チェック項目(3章) | 本ケース | 判定 |
|---|---|---|
| 3ホップ以上・可変長パスが主要クエリ | 人→PJ→技術→類似PJ→担当者 | ✅ |
| 関係の種類が増え続ける | スキル・案件・顧客・システム…拡大予定 | ✅ |
| 関係そのものに属性・分析価値 | 「担当した(役割・期間)」が重要 | ✅ |
| パターンマッチが要件 | 類似案件検索、影響分析 | ✅ |
| LLMに探索させたい | エージェント化の要望あり | ✅ |

- 5項目中5該当 → **Graph DB(Neo4j)採用**。ただし:
  - 人事マスタ・勤怠はRDBのまま(移行しない)
  - **併用アーキテクチャ**:RDB→KGへ同期(3章の移行戦略)

<!--
- 3章の演習でやった判定を「本番」で使う。チェックリストが実際に意思決定の道具として機能する場面を見せる
- 「全部✅だから採用」だけでなく、下段の「移行しないもの」が同じくらい重要。全面移行はしない、グラフ的クエリだけ切り出す — 3章の結論の実践
- 受講者への問いかけ:「もし✅が1個だけだったら?」→ RDBMS+再帰CTEで粘る、が正解。チェックリストは採用のためでなく判断のためにある
- 提案書ではこの表がそのまま「技術選定理由」の1ページになる、と総合課題への使い方を示す
-->

## Step 2:スキーマ設計 — CQ 10問から(5章)

コンピテンシークエスチョン(抜粋):

1. 「技術Xの経験者は誰か?」 2. 「PJ-Aに似た過去案件は?」
3. 「この障害に対応した経験があるのは?」 4. 「Sさんの知見はどの文書にあるか?」

→ 最小スキーマ:

```
(Person)-[:WORKED_ON {role, period}]->(Project)
(Project)-[:USES_TECH]->(Technology)
(Project)-[:FOR_CLIENT]->(Client)
(Document)-[:DESCRIBES]->(Project)
(Person)-[:AUTHORED]->(Document)
(Incident)-[:OCCURRED_IN]->(Project)
```

- CQに**出てこない**もの(勤怠、評価、給与)は**入れない**
- このスキーマがそのまま**LLM抽出用JSONスキーマ**(7章)になる

<!--
- 「CQ 10問がスキーマの仕様書」という5章の核心を実演。逆に、CQなしでスキーマを書き始めると際限なく肥大する
- 「入れない」判断を強調:給与や評価は技術的には入れられるが、CQにない+アクセス制御リスクが高い。スコープを絞る勇気が品質と信頼を守る
- WORKED_ONにrole/periodが付いている=「関係に属性」の実例。Step1のチェック項目と呼応していることを指差す
- 6章を思い出させる:TechnologyノードはWikidata IDでリンキングすると表記ゆれ(PostgreSQL/Postgres)に強くなる
-->

---

## Step 3:構築 — 7-8章のパイプラインを適用

| データ源 | 取り込み方法 | 根拠 |
|---|---|---|
| 人事マスタ・PJ管理DB(構造化) | **RDBから直接変換**(LLM不使用) | LLM不要な所に使わない(7章) |
| 議事録・日報・完了報告(非構造化) | **LLM抽出**(オントロジー誘導型) | 5章スキーマで制約 |

品質・コスト計画(8章):

- 名寄せ:社員はマスタID基準 / 技術名はembedding+LLM判定
- 品質目標:抽出 precision **90%** 以上(ゴールドデータ100文書で測定)
- コスト試算:文書2万件 × 平均3kトークン → 抽出は小型モデル+バッチAPI
- プロビナンス:全エッジに出典文書IDを付与(根拠提示・承認フローの基盤)

<!--
- 上段の表が7章の最重要原則の実践:「構造化データにLLMを通すのは品質もコストも悪化させるだけ」。人・PJの骨格はRDBから確実に作り、LLMは非構造化テキストからの肉付けに使う
- precision 90%という数字は「目標を先に決める」ことに意味がある。測る前に基準を決めないと、評価が「なんとなく良さそう」で終わる(8章)
- コスト試算をここで一度電卓レベルで見せると総合課題のROI試算の練習になる:2万件×3k×単価…と概算する姿を見せる
- 「ゴールドデータ100文書は誰が作る?」という問いを投げる:現場有識者を巻き込む=KGプロジェクトは技術だけでは完結しない
-->

---

## Step 4:活用 — 9-10章の使い分けで組む

```
ユーザー質問 → ルーター(質問タイプ判定)
   ├─ 単純な文書検索      → ベクトル検索(既存RAG流用)
   ├─ 構造的な質問        → ハイブリッドGraphRAG(9章)
   └─ 複数ステップの調査  → KGエージェント(10章)
```

| 経路 | 例 | レイテンシ/コスト |
|---|---|---|
| ベクトル | 「経費精算の手順は?」 | 低 |
| GraphRAG | 「技術Xの経験者は?」 | 中 |
| エージェント | 「A社障害の類似事例と対応可能な人材は?」 | 高 |

- 評価:質問セット50問で経路別に精度測定(9章のRagas評価を流用)

<!--
- 「全部GraphRAGにしない」「全部エージェントにしない」— 9章のルーティング設計がソリューション全体のコスト構造を決める
- 経路ごとのコスト差は体感で1:5:20くらいになる(エージェントはLLM呼び出しが多段)。安い経路で済む質問を高い経路に流さないだけでROIが大きく変わる
- ルーター自体は小型LLMや埋め込み分類で安価に作れる。凝りすぎない
- 「評価を流用」がポイント:9章で作った質問セットと評価パイプラインが資産になっている。評価は使い回せるように作る
- ここまでがケーススタディの設計編。次の2枚で一度立ち止まり「そもそも作るべきか」に戻る、と構成を予告
-->

## KGを作らない勇気(1):ベクトルRAGで十分な要件

| チェック | 該当したらベクトルRAG止まりでよい |
|---|---|
| 質問の答えが**単一文書内**で完結する(FAQ・マニュアル・規程) | ✅ |
| マルチホップ・集約・影響分析の質問が**ほぼ無い** | ✅ |
| 根拠は「該当文書の引用」で足りる(関係の経路は不要) | ✅ |
| エンティティ間の関係が**ビジネス上の関心事でない** | ✅ |

- ベクトルRAG:構築数日・追加文書は自動 / KG:スキーマ設計+品質管理が常時必要
- **まずベクトルRAGを作り、答えられない質問が溜まってからKGを検討**する順序も正当

> 1章の演習を思い出す:GraphRAGでも失敗する質問はあった。KGは万能ではない

<!--
- 最終章にこの話を置く理由:技術を学び切った直後が一番「使いたくなる」から。ハンマーを持つと全部釘に見える
- 判定の実務感覚:社内ヘルプデスクの質問ログを100件サンプリングして分類すると、構造的質問の比率が数字で出る。感覚でなくログで判定する
- 「後からKG」の順序が言い訳でなく戦略である理由:ベクトルRAGの失敗質問ログは、そのままKGのCQ(=何に答えるKGを作るか)の最良の素材になる
- 引用枠で1章の演習と接続:コース冒頭に撒いた伏線の回収であることを明示する
-->

## KGを作らない勇気(2):RDBMSで十分な要件

| チェック | 該当したらRDBMS(+Text2SQL)でよい |
|---|---|
| 探索が**1〜2ホップ固定**(JOINが2〜3回で済む) | ✅ |
| 集計・トランザクションが主体、グラフ的クエリは少数 | ✅ |
| スキーマが安定していて関係の種類が増えない | ✅ |
| データが既にRDBにあり、質問も定型的 | ✅ |

- その場合のLLM活用:**Text2SQL**+ビューで十分に「自然言語で聞ける」は実現できる
- 中間解も再確認(3章):再帰CTEで粘る/PostgreSQL+AGE
- 判断を誤ると:**KGの維持費(同期・品質・スキーマ進化)だけ払い続けて効果ゼロ**

<!--
- 前スライドとの違いを明確に:①は「そもそも構造が要らない」ケース、こちらは「構造は要るが既にRDBで足りている」ケース
- Text2SQLの実力は近年かなり高い。定型的な集計質問なら、KGを新設するよりRDBのスキーマ情報をLLMに渡す方が早くて安い
- 最下行が本音のリスク:KGは作った瞬間から同期・品質・スキーマ進化のランニングコストが発生する。効果がないKGは「負債」になる
- 3章の演習(自社要件の判定)をもう一度やってみるよう促す:コースを終えた今なら判定の解像度が上がっているはず
- 「作る」と決めた場合も次のスライドの「小さく作る」が続く、と繋ぐ
-->

## スモールスタート:CQ 10個から育てる

```
CQ 10個 → 最小スキーマ → 対象データを絞って構築 → ユーザーに使わせる
   ↑                                                     │
   └── 「答えられなかった質問」ログからCQを追加 ←────────┘
        (スキーマ拡張は開世界設計なので後から足せる:5章)
```

| フェーズ | 範囲 | 期間目安 |
|---|---|---|
| PoC | CQ10問・1部署・文書1,000件 | 2〜4週 |
| パイロット | CQ30問・3部署・評価とROI測定 | 1〜3ヶ月 |
| 本番 | 全社展開・エージェント・運用SLA | 継続 |

- 各フェーズの出口に**評価ゲート**:精度・利用数・時間短縮が基準未達なら拡大しない

<!--
- 「完璧なオントロジーを1年かけて作る」プロジェクトの対極。使われながら育つKGだけが生き残る
- ループ図の「答えられなかった質問ログ」が成長のエンジン。10章の探索ログ・9章の評価がここに流れ込む
- 「後から足せる」はグラフDBの構造的な強み(スキーマ変更にALTER TABLE的な痛みがない:3章・5章)。スモールスタートと技術特性が噛み合っている
- 評価ゲートを強調:スモールスタートは「小さく始める」だけでなく「基準未達なら止まる・やめる」を含む。やめる条件を先に決めるのが健全なPoC
- 受講者への問いかけ:「PoCの1部署はどこを選ぶ?」→ 痛みが強く協力的な部署。政治も設計のうち
-->

## ROIの示し方:効果を数字にする

| 効果 | 測り方 | 計算例(本ケース) |
|---|---|---|
| **回答精度の改善** | 質問セットで正答率A/B比較(9章) | 構造的質問:ベクトル62% → Graph 89% |
| **調査時間の短縮** | 対象業務の所要時間を前後測定 | 有識者探し 2h→10分 × 月40件 ≒ **73h/月** |
| **属人知識の資産化** | 引継ぎ工数・新人の立上り期間 | 引継ぎ1人あたり ▲20h、立上り ▲2週 |

**年間効果(時間短縮のみ)**:73h × 12ヶ月 × 5,000円/h ≒ **440万円/年**
**年間コスト**:構築120万(初年度)+ 運用60万 + LLM API 24万 ≒ **204万円/年**

> 初年度から効果>コスト。ただし**前提(件数・単価)を必ず明記** — 数字の誠実さが信頼を作る

<!--
- エンジニアが最も苦手にしがちな1枚だが、構造は単純:「効果=頻度×1件あたり削減×単価」の掛け算だけ
- 測定の要点:導入前のベースライン(現状2時間かかる)を先に測っておくこと。導入後に「前はどうだったか」は再現できない
- 精度改善は直接金額にしにくい → 「誤答による手戻り・意思決定ミス」の回避として定性+件数で語るのが現実的。無理に金額化して怪しい数字を作らない
- 属人知識の資産化はSさん退職の文脈で刺さる:「退職で消えるはずだった知見がKGとプロビナンス付き文書リンクで残る」
- 引用枠を最後に強調:盛った数字は一度バレると提案全体の信頼が死ぬ。前提を書いた控えめな数字の方が通る
-->

## グラフアルゴリズムの実務活用(Neo4j GDS)

KGは検索(RAG)だけでなく**分析資産**になる — Graph Data Science ライブラリ:

| アルゴリズム | 何が分かるか | 実務応用 |
|---|---|---|
| **中心性**(PageRank・媒介中心性) | ネットワークの要となるノード | キーパーソン特定・退職リスク影響・ボトルネック |
| **類似度**(Node Similarity) | 繋がり方が似たノード | 類似案件推薦・「この人と近いスキルの人」 |
| **コミュニティ検出**(Louvain) | 密に繋がるグループ | 組織の実態把握・**不正リング検出**・知識クラスタ |
| **経路探索** | 最短・全経路 | 影響伝播分析・依存関係追跡 |

```cypher
CALL gds.pageRank.stream('knowledge-graph')
YIELD nodeId, score  // 「最も多くの知識が集中する人」ランキング
```

- LLM連携:アルゴリズム結果をノードに書き戻し → GraphRAGの検索・回答文脈に活用

## 運用設計:作った後の方が長い

| 項目 | 設計ポイント |
|---|---|
| **RDBとの継続同期** | バッチ(夜間・冪等)から始め、鮮度要件が上がったらCDCへ(3章) |
| **スキーマ進化** | 追加は容易(開世界)。**リネーム・意味変更は移行手順+抽出プロンプト側の更新**を忘れずに |
| **アクセス制御** | ロールベース+サブグラフ単位(部署ラベルで制限)。エージェントは読取専用(10章) |
| **バックアップ** | 定期dump+**抽出パイプラインの再実行可能性**も復旧手段(7章の冪等設計) |
| **監視** | クエリ遅延・ノード/エッジ数の推移・抽出パイプライン失敗率・LLM APIコスト |
| **鮮度SLA** | 例:「新規文書は24h以内に反映」「マスタ変更は翌朝反映」を明文化 |

<!--
- 「動くものができた」はプロジェクトの折り返し地点にすぎない。運用コストがROI計算の分母に入っていたことを思い出させる
- スキーマ進化の落とし穴を具体的に:リレーション名を変えると、7章の抽出プロンプト・9章のText2Cypher・10章のツールdescriptionが全部影響を受ける。スキーマは「KGとLLMの契約書」なので変更は契約変更として扱う
- 「パイプラインの再実行可能性も復旧手段」は見落とされがちな観点:原文書+冪等パイプラインがあればKGは再構築できる。原文書とプロビナンスこそ最重要バックアップ対象
- 鮮度SLAは8章の鮮度管理の運用への落とし込み:「いつのデータか」に答えられないKGは信頼されず使われなくなる
- 監視にLLM APIコストが入っている点:エージェント(10章)は放っておくとコストが動く。上限アラートは必須
-->

## 性能とスケール

| 観点 | 見積もり・対策 |
|---|---|
| **データ量見積もり** | ノード数≒エンティティ種×件数、エッジは通常ノードの5〜20倍。embedding格納は 1ノードあたり数KB → メモリ設計に効く |
| **クエリチューニング** | `PROFILE`でdb hits確認(4章)。起点ノードにインデックス必須。可変長パスは**深さ上限**(`*1..3`)を必ず付ける |
| **書き込みスケール** | バルクロードは `LOAD CSV`/バッチMERGE。抽出パイプラインは並列化 |
| **LLM律速への対策** | 実は**ボトルネックはDBよりLLM**になりがち:キャッシュ(同一質問・スキーマ取得結果)、小型モデルへのルーティング、並列呼び出し、ストリーミング応答 |

> 目安:数百万ノード規模まではモデリングとインデックスの問題。分散が必要になる前に設計を疑う

<!--
- 順番が大事:性能問題の9割は「スケールアウトが必要」ではなく「モデリングかインデックスが悪い」。4章のPROFILEに立ち返るのが第一手
- 可変長パスの深さ無制限(`*`)は事故の定番。10章のエージェントが生成するクエリにも効くので、ガードレール側でも上限を強制する
- 「LLM律速」は体感と違うので強調:Neo4jのクエリは数十ms、LLM呼び出しは数秒。応答時間のグラフを見せるとDBを疑っていた受講者が驚く
- スキーマ取得結果のキャッシュは効果絶大で実装が簡単(スキーマは分単位では変わらない)。エージェントの毎回のget_schema呼び出しをキャッシュで返す
- embedding容量の話は4章のベクトルインデックスの運用面:次元数×ノード数で線形に効く
-->

## よくある失敗パターン集

| 失敗パターン | 症状 | 回避策(本コースの該当章) |
|---|---|---|
| **完璧なオントロジーを目指して頓挫** | 半年設計して1件も質問に答えていない | CQ10個の最小スキーマで始める(5章・本章) |
| **全データ投入で品質崩壊** | ノイズだらけで検索結果が信用されない | スコープを絞る+品質評価ゲート(8章) |
| **評価なしのデモ止まり** | 「すごい」で終わり本番に進まない | 質問セット+定量比較+ROI(9章・本章) |
| **グラフ化自体が目的化** | 「KGはできた。で、何に使うんだっけ」 | CQ起点の設計、作らない勇気(3章・本章) |
| **書き込み野放しで汚染** | エージェント/抽出の誤りが増殖 | 承認フロー+プロビナンス(8章・10章) |

> 共通の根:**「何の質問に答えるためのKGか」を見失うこと**

- 4つの失敗は独立ではなく連鎖する:目的化→全データ投入→品質崩壊→評価もできず→デモ止まり、という典型コース

## 第11章まとめ — そしてコースの結び

- 設計は**CQ(何の質問に答えるか)から始め**、各章の道具で意思決定を積む:
  3章選定 → 5章スキーマ → 7-8章構築 → 9-10章活用 → 運用・ROI
- **作らない勇気**:ベクトルRAG・RDBMSで足りるなら、それが正解
- 効果は数字で:**精度A/B・時間短縮×頻度×単価・前提の明記**
- 失敗パターンの根は共通:**目的(CQ)を見失うこと**
- KGは検索を超えて**分析資産**になる(GDS)— そして運用してこそ資産

> **LLMとKnowledge Graphを組み合わせ、効率的に効果が出るソリューションを作る** —
> このコースの真のゴールを、明日から自分の現場で。
