# レクチャーコース:LLM × Knowledge Graph — 効果が出るソリューションを作る

## コース概要

| 項目 | 内容 |
|---|---|
| 対象者 | RDBMSの知識・プログラミング経験・生成AIプログラミング経験を持つITエンジニア |
| 真のゴール | **LLMとKnowledge Graphを組み合わせ、効率的に効果が出るソリューションを設計・実装・評価できるエンジニアを育てる** |
| 到達点 | ①Graph DBを使うべき要件を判断できる ②LLMでKGを構築できる ③GraphRAG・エージェント連携を実装できる ④品質・コスト・ROIを評価できる |
| 構成 | 全11章 + 付録(各章60〜90分想定、講義+ハンズオン演習) |
| 技術スタック | Neo4j / Cypher を主軸、Python + LangChain / LlamaIndex / Neo4j GraphRAG。RDF / SPARQL は外部知識活用の手段として1章 |

### コースの設計思想

「セマンティックウェブの体系」ではなく「LLMソリューションの実装」を背骨に据える。第1章で動くGraphRAGを見せて目的地を示し、以降の章はすべて「そのソリューションを支える技術」として学ぶ。OWL推論器・Protégé・R2RML等の古典的セマンティックウェブ技術の深掘りは付録に退避。評価・コスト・「KGが不要なケースの見極め」を本編に組み込む。

## 第1章 なぜLLMにはKnowledge Graphが必要か

**学習目標**:ベクトルRAGの限界を実例で体感し、KGが解決する問題領域とコース全体の目的地を把握する。

**内容**:
- LLM実務の3大課題:ハルシネーション、根拠提示、マルチホップ推論の弱さ
- 【ライブデモ】ベクトルRAGが失敗する質問を見る:「AさんとBさんの共通の取引先は?」「この障害の影響範囲は?」— チャンク検索では答えられない構造的質問
- 【ライブデモ】同じ質問がGraphRAGでは答えられることを確認(この時点では中身はブラックボックスでよい)
- Knowledge Graphとは何か:エンティティと関係で知識を表す、Google Knowledge Graph、実ユースケース(不正検知、影響分析、社内ナレッジ統合)
- 用語の交通整理:Ontology / Knowledge Graph / Graph DB / RDF の関係マップ(詳細は各章で)
- 「LLM×KG」の組み合わせパターン全体像:KG構築をLLMで自動化(7-8章)/ KGでLLMの回答を強化(9章)/ KGをエージェントのツールに(10章)
- 環境構築ガイダンス:Docker で Neo4j、Python + LLM API

**演習**:配布のGraphRAGデモ環境を動かし、ベクトルRAGとの回答差を数問で比較する。

## 第2章 グラフデータモデルの基礎

**学習目標**:プロパティグラフモデルを理解し、RDBスキーマとの対応関係でグラフモデリングを説明できる。

**内容**:
- グラフ理論の最低限の語彙:ノード、エッジ、有向グラフ、パス、次数、トラバーサル
- **プロパティグラフモデル**:ラベル付きノード、型付きリレーション、プロパティ — 本コースの標準モデル
- RDBスキーマとの対応:テーブル→ノードラベル、行→ノード、外部キー→リレーション、中間テーブル→リレーション(+プロパティ)
- グラフモデリングの基本判断:ノードにするか、プロパティにするか、リレーションにするか
- もう一つのモデル「RDFトリプル」の概要:主語–述語–目的語(詳細は6章。ここでは両者の位置づけのみ)
- LLMとの親和性:グラフ構造は「エンティティと関係」というLLMの抽出出力と自然に対応する — 後半章への伏線
- グラフアルゴリズムの入り口:最短経路、中心性、コミュニティ検出(活用は11章)

**演習**:サンプルRDBスキーマ(ECサイト:顧客・注文・商品)をプロパティグラフにモデリングし、設計判断を言語化する。

## 第3章 RDBMSでグラフを扱う限界と Graph DB を使うべき要件

**学習目標**:「RDBMSによるグラフ表現で粘るべきか、Graph DBに移行すべきか」を要件ベースで判断できる。

**内容**:
- RDBMSでのグラフ表現パターンの棚卸し:隣接リスト、経路列挙、閉包テーブル、入れ子集合 — それぞれの得意・不得意
- 再帰CTE(`WITH RECURSIVE`)によるグラフ探索:書き方と、深さ・分岐が増えたときに起きること
- 性能特性の本質的な違い:JOINのコスト(インデックス参照の繰り返し)vs index-free adjacency(ポインタ辿り)— 深さNの探索での計算量比較を実測で見る
- 性能以外の限界:スキーマ変更コスト(関係の種類が増えるたびのALTER/中間テーブル追加)、クエリの可読性・保守性、可変長パス・パターンマッチの表現不能性
- **Graph DBを使うべき要件チェックリスト**:
  - 3ホップ以上の探索・可変長パスが主要クエリにある
  - 関係の種類が増え続ける/事前に確定できない
  - 「関係そのもの」に属性や分析価値がある
  - パターンマッチ(不正検知、影響分析)が要件にある
  - LLMに探索させたい(エージェントがスキーマを理解して自律的にクエリする)
- **RDBMSで十分なケース**:探索が1〜2ホップ固定、集計・トランザクションが主体、グラフ的クエリが全体の少数 — 「なんでもグラフ」アンチパターンの回避
- 中間選択肢:RDBMS のグラフ拡張(PostgreSQL + Apache AGE 等)、両用DBの現実
- 移行戦略:全面移行ではなく「グラフ的クエリだけGraph DBに切り出す」併用アーキテクチャ、RDBからの同期パターン(バッチ/CDC)
- 【実測ハンズオン】同一データ(組織階層+取引ネットワーク)でPostgreSQL再帰CTEとNeo4jの探索性能・クエリ記述量を比較

**演習**:自社(または想定)システムのクエリ要件をチェックリストに当てはめ、移行対象・併用構成を提案書1枚にまとめる。

## 第4章 Neo4j / Cypher 実践

**学習目標**:Neo4jでデータモデリング・CRUD・基本的な性能チューニングができる。

**内容**:
- Neo4jエコシステム:Desktop / AuraDB / Browser、他のプロパティグラフDB(Amazon Neptune, Memgraph)と標準化動向(GQL)
- **Cypher基礎**:`MATCH` / `CREATE` / `MERGE` / `SET` / `DELETE`、パターン記法 `(a)-[:REL]->(b)`
- SQL→Cypher対応表で最短習得:JOIN→パターンマッチ、GROUP BY→集約、再帰CTE→可変長パス `[:REL*1..3]`
- 最短経路と可変長パス、`shortestPath`
- データモデリング実践:中間ノードパターン、時系列の扱い、リレーション方向の設計
- インデックスと制約、`EXPLAIN` / `PROFILE` によるチューニング
- **ベクトルインデックス**:Neo4jにembeddingを格納して類似検索する(9章のハイブリッドGraphRAGの土台)
- Pythonからの操作:公式ドライバ、バルクロード(`LOAD CSV`)

**演習**:2章でモデリングしたECデータをNeo4jに投入し、推薦クエリ(共同購買)と影響分析クエリを実装する。

## 第5章 LLM時代のオントロジー設計 — スキーマでLLMを制御する

**学習目標**:オントロジーを「LLMの抽出・生成を制約する設計図」として理解し、実用的なグラフスキーマを設計できる。

**内容**:
- オントロジーとは:「概念とその関係の明示的な仕様」— 哲学的背景は最小限に、エンジニアリングの道具として
- **LLM時代のオントロジーの実用価値(本章の核心)**:
  - LLM抽出のスキーマ制約:structured output のJSONスキーマとして機能し、抽出のブレを抑える
  - Text2Cypher への入力:スキーマ情報の質がクエリ生成精度を決める
  - 用語の正規化辞書:同義語・表記ゆれをクラスと関係に収束させる
- RDBスキーマ設計との違い:閉世界 vs 開世界、後から関係を足せる設計
- スキーマ設計の実践手順:コンピテンシークエスチョン(このKGは何の質問に答えるべきか)から始める、命名規約、粒度の決め方
- 既存語彙の再利用:schema.org を「車輪の再発明を防ぐカタログ」として使う、ドメイン語彙(FIBO、SNOMED等)の存在を知る
- クラス階層・関係の制約(domain/range、カーディナリティ)の実務的な使いどころ — OWL公理・推論器の深掘りはしない(付録参照)
- アンチパターン:過度な抽象化、哲学的正しさの追求、使われない語彙の量産
- LLMにオントロジー設計を手伝わせる:ドラフト生成→人間がレビューのワークフロー

**演習**:社内ナレッジ(人・組織・スキル・ドキュメント)のコンピテンシークエスチョンを定義し、グラフスキーマ+LLM抽出用JSONスキーマに落とす。

## 第6章 RDF / SPARQL と外部知識の活用

**学習目標**:RDFの読み書きとSPARQLの基本を習得し、Wikidata等の公開ナレッジグラフを自社KGの強化に使える。

**内容**:
- RDFを学ぶ実務的理由:公開ナレッジグラフ(Wikidata, DBpedia)と標準語彙の世界はRDFでできている — 「教養」ではなく「外部知識の取り込み手段」
- RDF最速理解:トリプル(主語–述語–目的語)、IRI、リテラル、Turtle記法の読み書き、JSON-LD(Web API連携で登場する形式)
- **SPARQL基礎**:`SELECT` / `WHERE`、`FILTER`、`OPTIONAL`、集約 — SQL対比で最短習得
- プロパティパスによる多段探索
- 【ハンズオン】Wikidata SPARQLエンドポイント:企業・人物・地理などの実データ取得
- 外部KGの活用パターン:エンティティリンキングの参照先(8章で使用)、マスタデータの補完、業界標準IDへの接続
- RDF ⇄ プロパティグラフの変換:考え方と実装(neosemantics)
- 深掘りしないと宣言するもの:OWL推論、SHACL、フェデレーテッドクエリ、トリプルストア構築(必要になったら付録・参考資料へ)

**演習**:Wikidataから日本のIT企業データをSPARQLで取得し、Neo4jの自社KGにマージする。

## 第7章 LLMによるKG自動構築(1):抽出パイプライン

**学習目標**:LLMで非構造化テキストからトリプルを抽出し、Neo4jに投入するパイプラインを実装できる。

**内容**:
- KG構築の全体像:スキーマ設計(5章)→ 抽出 → 正規化・解決(8章)→ 投入 → 品質管理(8章)
- なぜLLMで構築するのか:従来のNER/関係抽出パイプラインとの精度・開発コスト比較、使い分け
- LLMトリプル抽出の設計:プロンプト設計、structured output(JSONスキーマ=5章のオントロジー)、few-shot例の作り方
- オントロジー誘導型抽出:許可するエンティティ型・関係型を制約し、ハルシネーションを抑える
- チャンク戦略:文書分割と文脈保持、抽出漏れ・重複への対処
- 既製ツールの活用と限界:LangChain `LLMGraphTransformer`、LlamaIndex、Neo4j GraphRAG パッケージ(`SimpleKGPipeline`)— 中身を理解した上で使う
- 構造化データの取り込み併用:RDB・CSVからの直接変換(LLM不要な部分にLLMを使わない)
- パイプラインとしての実装:冪等性、再実行、差分更新の設計

**演習**:ニュース記事・社内文書風テキストからLLMでトリプルを抽出し、Neo4jに投入・可視化する。

## 第8章 LLMによるKG自動構築(2):エンティティ解決・品質・コスト

**学習目標**:抽出したKGの品質を測定・改善し、コストを見積もれる — 「動くデモ」と「使えるシステム」の分水嶺。

**内容**:
- **エンティティ解決(名寄せ)**:「トヨタ」「トヨタ自動車」「Toyota Motor」を同一ノードに — ルールベース、embedding類似度、LLM判定の組み合わせ
- 外部KGへのリンキング:Wikidata ID等の安定識別子で正規化(6章の応用)
- **品質評価**:抽出のprecision/recallの測り方、評価用ゴールドデータの作り方、LLM-as-a-judgeの活用と限界
- ハルシネーション対策:原文根拠(プロビナンス)をエッジに持たせる、確信度スコア、人間レビューのワークフロー設計
- グラフ品質の検証:スキーマ違反チェック、孤立ノード・重複エッジの検出、品質ダッシュボード
- **コスト設計**:トークン量見積もり、モデル選定(抽出は小型モデル+検証は大型モデル等の段階戦略)、キャッシング、バッチAPI
- 継続更新の設計:新規文書の増分処理、エンティティの経時変化、KGの鮮度管理

**演習**:7章のパイプラインに名寄せと品質評価を組み込み、precision/recallとAPIコストをレポートする。

## 第9章 GraphRAG:アーキテクチャ類型と実装

**学習目標**:GraphRAGの主要アーキテクチャを使い分け、ベクトルRAGと定量比較できる。

**内容**:
- ベクトルRAGの限界の構造的理解(1章のデモの種明かし):チャンク分断、マルチホップ不能、集約質問への弱さ
- **GraphRAGアーキテクチャ4類型と使い分け**:
  - Text2Cypher:自然言語→クエリ変換。スキーマの与え方、クエリ検証、誤生成対策
  - グラフ探索型:質問からエンティティ抽出→近傍サブグラフ取得→コンテキスト化
  - ハイブリッド:ベクトル検索でエントリポイント発見→グラフ展開(4章のベクトルインデックス活用)— 実務の第一選択
  - コミュニティ要約型(Microsoft GraphRAG):全体俯瞰質問への対応、インデックス構築コストとの引き換え
- 実装ハンズオン:LangChain / Neo4j GraphRAG パッケージによるQAシステム(ハイブリッド構成)
- **評価**:faithfulness / answer relevancy 等のRAG評価指標、Ragas等による自動評価、ベクトルRAGとのA/Bベンチマーク
- 質問タイプ別ルーティング:単純検索はベクトル、構造的質問はグラフ — 全部GraphRAGにしない設計
- レイテンシとコスト:クエリ経路ごとの応答時間、キャッシュ

**演習**:7-8章で構築したKGに対しハイブリッドGraphRAGを実装し、ベクトルRAGと同一質問セットで定量比較レポートを作成する。

## 第10章 エージェント × KG:自律的にグラフを使うLLM

**学習目標**:KGをツールとして使うLLMエージェントを設計・実装できる。

**内容**:
- QAボットからエージェントへ:単発の質問応答ではなく、複数ステップの調査・判断タスクをKGで支える
- KGをエージェントのツールにする設計:スキーマ取得ツール+クエリ実行ツール、探索の逐次計画(一発Text2Cypherとの違い)
- **MCPによるグラフDB接続**:Neo4j MCPサーバ、ツール定義の設計、権限とガードレール(書き込み制御、クエリコスト制限)
- エージェントの信頼性設計:クエリ失敗時のリトライ・自己修正、探索の発散防止(ステップ上限、コスト上限)
- KGへの書き戻し:エージェントが得た新事実をKGに追記する(人間承認フロー付き)— 自己成長するナレッジベース
- マルチエージェント構成:抽出エージェント+検証エージェント+QAエージェントの分業
- 実装ハンズオン:MCP経由でNeo4jに接続したエージェントが、マルチホップの調査質問(「この障害の影響を受ける顧客と担当営業は?」)を自律的に解く

**演習**:自社KGに対する調査エージェントを実装し、探索ログから動作を分析・改善する。

## 第11章 総合実践:ソリューション設計・評価・ROI

**学習目標**:要件からDB選定・設計・実装・評価・運用まで一連の意思決定を行い、効果を示せる。

**内容**:
- ケーススタディ:社内ナレッジグラフ+GraphRAG+エージェントをゼロから設計(要件定義→3章チェックリストでDB選定→5章スキーマ→7-8章構築→9-10章活用)
- **「効果が出る」ための設計判断**:
  - KGを作らない勇気:ベクトルRAGで十分な要件、RDBMSで十分な要件の再確認
  - スモールスタート:コンピテンシークエスチョン10個から始め、使われるKGを育てる
  - ROIの示し方:回答精度改善・調査時間短縮・属人知識の資産化を数字にする
- グラフアルゴリズムの実務活用:Neo4j GDSによる中心性・類似度・コミュニティ検出 — 推薦、不正検知、組織分析への応用
- 運用設計:RDBとの継続同期、スキーマ進化、アクセス制御、バックアップ、監視、KG鮮度のSLA
- 性能とスケール:データ量見積もり、クエリチューニング、LLM呼び出しの律速対策
- よくある失敗パターン集:完璧なオントロジーを目指して頓挫、全データ投入で品質崩壊、評価なしのデモ止まり、グラフ化自体が目的化
- コース総括と学習ロードマップ

**演習(総合課題)**:ケーススタディのソリューションを完成させ、技術選定理由・評価結果・ROI試算を含む提案書としてプレゼンする。
