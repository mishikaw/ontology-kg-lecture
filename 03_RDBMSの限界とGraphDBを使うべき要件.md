## 現場の現実:RDBMSの中に潜むグラフ

みなさんのデータベースにこんなものはありませんか?

| よくある実装 | 隠れているグラフ |
|---|---|
| `employee.manager_id`(自己参照FK) | 組織階層 |
| `parts_hierarchy(parent_id, child_id)` | 部品構成(BOM) |
| `system_dependencies(from_id, to_id)` | システム依存関係 |

- 悪い設計ではない。**要件が軽いうちは合理的な選択**
- 問題は「どこから苦しくなるか」を知らずに粘り続けること

## RDBMSでのグラフ表現パターンの棚卸し

| パターン | 仕組み | 得意 | 不得意 |
|---|---|---|---|
| **隣接リスト** | `parent_id` 列 | 書き込み、1ホップ | 深い探索(再帰CTE必須) |
| **経路列挙** | パス文字列 `/1/4/7/` | 祖先の取得(LIKE) | 整合性維持、パス更新 |
| **閉包テーブル** | 全祖先×子孫ペアを格納 | 部分木の読み取り | 書き込み、容量 O(n²) |
| **入れ子集合** | left/right 番号 | 部分木の読み取り | 挿入(大量更新)、**木以外不可** |

- どれも「**読みと書きのトレードオフ**」の別解にすぎない
- そして全パターン共通の弱点:**ネットワーク型(DAG・循環あり)への対応**と
  **関係の種類が増えたときの拡張性**

<!--
- SQLアンチパターン(Bill Karwin)のナイーブツリー章と同じ整理。既読の人がいれば接続してあげる
- 各パターンを30秒ずつ:経路列挙は「材料パス方式」、入れ子集合は「入れ子の括弧に番号を振る」と補足。深入りはしない(次スライドから隣接リスト+閉包テーブルに絞る)
- 「木ならこの4つで戦える。しかし実世界のグラフは木ではない」が最重要メッセージ。組織の兼務、部品の共通化、相互依存 — すぐDAGや循環が出てくる
- 質問例:「兼務(上司が2人)が発生した瞬間、入れ子集合はどうなりますか?」→ 破綻。モデルの前提を確認する癖をつけさせる
-->

## 隣接リスト:最も一般的なパターン

```sql
CREATE TABLE employee (
  id         INT PRIMARY KEY,
  name       TEXT NOT NULL,
  manager_id INT REFERENCES employee(id)   -- 自己参照FK
);

-- 1ホップ(直属の部下)は簡単で速い
SELECT * FROM employee WHERE manager_id = 42;

-- 2ホップは自己JOIN
SELECT e2.* FROM employee e1
JOIN employee e2 ON e2.manager_id = e1.id
WHERE e1.manager_id = 42;
```

- 書き込みは1行の更新で済む。**固定の浅いホップ数**なら十分実用的
- 問題は「**深さが決まらない探索**」:「配下の全員」はJOINを何回書く?

<!--
- 全員が書いたことのあるSQLから始めて安心感をつくる
- 「JOINを深さの回数だけ書く」やり方の破綻を口頭で:深さが分からない、10段組織なら10回自己JOIN、UNIONで全深さを束ねる…書けなくはないが保守不能
- 「深さが決まらない」=第2章の語彙で言う可変長パス。この語を再登場させて接続する
- ここで再帰CTEの存在を知っている人を確認。知っていれば「次で復習+限界の話」、初めてなら「これから武器を1つ増やす」と調整
-->

## 閉包テーブル:読みを買って、書きを売る

```sql
CREATE TABLE org_closure (
  ancestor_id   INT,
  descendant_id INT,
  depth         INT,
  PRIMARY KEY (ancestor_id, descendant_id)
);  -- 全ての祖先→子孫ペアを事前に格納(自分自身 depth=0 も)

-- 配下全員が再帰なしの1クエリ!
SELECT descendant_id FROM org_closure
WHERE ancestor_id = 42 AND depth > 0;
```

| コスト | 内容 |
|---|---|
| 容量 | 最悪 O(n²) 行(深い組織ほど爆発) |
| 書き込み | 1人の異動で**大量ペアの削除・再挿入** |
| 拡張性 | **関係の種類ごとに閉包テーブルがもう1つ**必要 |

<!--
- 「読み取り性能はマテリアライズで買える」という一般原則の好例。読みは確かに速い、と正当に評価する
- 異動の更新コストを具体的に:部長が部ごと他事業部に移ると、配下全員×全祖先のペアを貼り替え。トランザクションが巨大化しロック競合の温床に
- 最後の行が本章の伏線:「report_to用、担当用、依存用…閉包テーブルが増殖していく。関係の種類が増えるシステムでは維持不能」→ チェックリスト項目2に繋がる
- 「事前計算で読みを買う」はGraph DBの index-free adjacency と発想が同じ、という指摘は鋭い受講者から出ることがある。「その通り。ただしGraph DBは書き込み時のペナルティがO(1)」と返す
-->

## 再帰CTE(`WITH RECURSIVE`)によるグラフ探索

```sql
WITH RECURSIVE subordinates AS (
  -- ① 起点(非再帰項)
  SELECT id, name, 1 AS depth, ARRAY[id] AS path
  FROM employee WHERE id = 42
  UNION ALL
  -- ② 1ホップ拡張を繰り返す(再帰項)
  SELECT e.id, e.name, s.depth + 1, s.path || e.id
  FROM employee e
  JOIN subordinates s ON e.manager_id = s.id
  WHERE s.depth < 5
    AND NOT e.id = ANY(s.path)   -- 循環対策(自前!)
)
SELECT * FROM subordinates;
```

- 標準SQLで可変長探索が書ける、RDBMS側の正攻法
- ただし:**深さ上限・循環検出・経路記録をすべて自前で管理**

<!--
- ライブで1行ずつ動かして見せたい。①が種、②が「前ラウンドの結果に1ホップJOIN」という実行モデルを図解する
- path配列による循環対策はPostgreSQL方言。これを忘れて本番で無限ループさせた事故談があれば話す(鉄板のつかみ)
- 「エンジン任せにできない」ことに注目:探索という操作の制御(深さ、訪問済み、経路)が全部アプリケーションロジックとしてSQLに漏れ出している
- とはいえ武器としては優秀:「月次で1回、深さ5まで」のような要件ならこれで十分。過小評価もしない
-->

## 深さ・分岐が増えると何が起きるか

| 現象 | 中身 |
|---|---|
| **中間結果の爆発** | 各ラウンドで前結果×平均分岐数に増殖。分岐10・深さ5 → 10万行規模の作業テーブル |
| **毎ホップがJOIN** | ラウンドごとにインデックス検索を全行分繰り返す |
| **循環・重複訪問** | path配列の比較コストが行数×深さで効いてくる |
| **オプティマイザの限界** | 再帰の結果行数を事前推定できず、実行計画が安定しない |
| **複数の関係種類** | 「REPORTS_TO と ASSIGNED_TO を跨いで辿る」— 再帰項がUNIONだらけの怪物SQLに |

> 深さ2〜3・単一関係なら実用的。
> **深さ4以上・複数関係・対話的応答**が要件になった時が曲がり角

<!--
- 前スライドのSQLがどう苦しくなるかの答え合わせ。b^d(分岐^深さ)の指数的増殖を板書すると直感に刺さる
- 「複数の関係種類」が実務で最初に効いてくる限界であることが多い:影響分析は システム→サービス→顧客→担当者 と関係の種類を乗り換える。単一テーブルの再帰では書けない
- オプティマイザの話は深入りしない。「統計が効かない世界に入る」という一言で十分
- 最後の枠の3条件(深さ4+、複数関係、対話的)がチェックリストの前振り。「自分のクエリはどっち側か」を考えさせてから次へ
-->

## 本質的な違い:JOIN vs index-free adjacency

```
RDBMS(JOIN):関係はクエリ時に「計算」する
  行 → FK値 →〔インデックス検索 O(log N)〕→ 行 → …を毎ホップ繰り返す
        ※ N = テーブル全行数。データが増えるほど1ホップが遅くなる

Graph DB(index-free adjacency):関係は格納時に「実体」として持つ
  ノード →〔隣接ポインタを直接辿る O(1)〕→ ノード → …
        ※ コストは実際に触った近傍の数だけ。全体データ量 N と無関係
```

| | 1ホップのコスト | 深さ d の探索(訪問 k ノード) |
|---|---|---|
| RDBMS | O(log N) × 行数 | **O(k log N)** — Nに引きずられる |
| Graph DB | O(1) × 近傍数 | **O(k)** — 触った分だけ |

<!--
- 本章の理論的な核。「インデックスを引く」対「ポインタを辿る」の一言に凝縮して何度も繰り返す
- 直感的な比喩:RDBMSは毎回電話帳で番号を調べて掛ける、Graph DBは短縮ダイヤルが登録済み。友達10人に順に繋ぐとき、電話帳方式は町の人口が増えるほど遅くなる
- 「index-free」の意味の注意:起点ノードを見つける最初の1回はインデックスを使う。フリーなのは「ホップごとの隣接参照」
- 公平性のための一言:このトレードオフの代償としてグラフDBは任意列の突き合わせ(アドホックJOIN)や大規模集計がRDBより苦手。万能ではない
- 次のスライドで「では実際どのくらい差が出るか」の数字を見る
-->

## 実測イメージ:深さNの探索(friends-of-friends)

100万ノードのソーシャルグラフで「友達の友達の…」を深さを変えて探索:

| 探索深さ | RDBMS | Neo4j | 返る人数(概数) |
|---|---|---|---|
| 2 | 0.016 秒 | 0.01 秒 | 約 2,500 |
| 3 | 30.2 秒 | 0.168 秒 | 約 11 万 |
| 4 | **1,543 秒** | 1.36 秒 | 約 60 万 |
| 5 | **未完了** | 2.13 秒 | 約 83 万 |

<span class="small">出典:Robinson, Webber, Eifrem『Graph Databases 2nd ed.』(O'Reilly)の実験。数値は環境依存の参考値</span>

- 深さ2ではほぼ互角 → **深さ3で3桁の差** → 深さ4で「秒 vs 20分超」
- 本章末の**実測ハンズオン**で自分の環境でも確かめる

<!--

## 性能以外の限界(1):スキーマ変更コスト

**「新しい種類の関係」を追加したくなったら?**(例:社員間の「メンター」関係)

| | RDBMS | Graph DB |
|---|---|---|
| スキーマ作業 | 中間テーブル追加 or 列追加(ALTER) | **不要**(新しい型のエッジを張るだけ) |
| データ移行 | マイグレーション計画・実施 | 不要 |
| 既存クエリ | JOIN追加の改修が波及 | 既存パターンは無影響 |
| 探索クエリ | 再帰CTEを関係ごとに書き直し | `[:MENTORS\|:REPORTS_TO*]` で混在探索 |

- KGの実務では**関係の種類は増え続ける**(LLM抽出なら特に:7章)
- 「関係の追加が定例作業」のシステムで、ALTERとマイグレーションは重い税金

<!--
- 性能の話に隠れがちだが、実務ではこちらが先に効く現場も多い。「先週も中間テーブルを足した」経験を聞いてみる
- Graph DB側の柔軟性の裏面も必ず言う:統制なしに型が増えると命名の揺れ(MENTORS/mentor_of/指導)で汚染される。自由には設計規律が必要 → 5章のオントロジーがその答え
- LLM抽出との関係が本コース的に重要:抽出のたびに新種の関係が発生しうる世界では、スキーマ固定コストが致命的になる
- 「事前に確定できない関係」というフレーズをチェックリスト項目2の言葉として先出ししておく
-->

## 性能以外の限界(2):クエリの可読性・表現力

<div>

**SQL(再帰CTE)— 約15行**

```sql
WITH RECURSIVE subs AS (
  SELECT id, name, 1 AS depth,
         ARRAY[id] AS path
  FROM employee WHERE id = 42
  UNION ALL
  SELECT e.id, e.name, s.depth+1,
         s.path || e.id
  FROM employee e
  JOIN subs s
    ON e.manager_id = s.id
  WHERE s.depth < 5
    AND NOT e.id = ANY(s.path)
)
SELECT name FROM subs;
```

</div>
<div>

**Cypher — 2行**

```cypher
MATCH (m:Employee {id:42})
      <-[:REPORTS_TO*1..5]-(s)
RETURN s.name
```

- **可変長パス** `*1..5` (深さ・循環・経路はエンジン管理)
- **最短経路** `shortestPath(...)`
- **循環パターン検出**(不正リング)
- 関係の種類を跨ぐパターンマッチ

</div>
</div>

## Graph DB を使うべき要件チェックリスト

| # | 要件 | 該当? |
|---|---|---|
| 1 | **3ホップ以上の探索・可変長パス**が主要クエリにある | ☐ |
| 2 | **関係の種類が増え続ける**/事前に確定できない | ☐ |
| 3 | **「関係そのもの」に属性や分析価値**がある(取引額、確信度、期間…) | ☐ |
| 4 | **パターンマッチ**(不正検知、影響分析、推薦)が要件にある | ☐ |
| 5 | **LLMに探索させたい**(エージェントがスキーマを理解して自律クエリ) | ☐ |

> 目安:**2つ以上該当 → Graph DB(または併用)を本格検討**
> 1にも該当し、かつ対話的な応答時間が必要なら性能面だけでも移行事由になる

## RDBMSで十分なケース — なんでもグラフ化しない

| 状況 | 判断 |
|---|---|
| 探索が**1〜2ホップ固定**(JOIN 2回で書ける) | RDBのJOINで十分速い |
| **集計・トランザクションが主体**(会計、在庫、決済) | RDBの独壇場。移行は劣化 |
| グラフ的クエリが**全体のごく少数**(月次バッチ1本など) | 再帰CTEで粘るのが合理的 |
| チームの**運用資産**(監視、バックアップ、スキル)がRDB前提 | 移行コストが利益を上回りがち |

**アンチパターン:「なんでもグラフ化」**
- 売上集計をグラフでやる、マスタ管理をグラフに移す → SQLの方が速くて簡単
- 新技術の採用自体が目的化 → 11章「KGを作らない勇気」の先取り

> DB選定は要件で決める。**「グラフを使いたい」は要件ではない**

## 中間選択肢:RDBMSのグラフ拡張

| 選択肢 | 概要 | 向く場面 |
|---|---|---|
| **PostgreSQL + Apache AGE** | 拡張機能でopenCypherが使える。SQLと同一DB内で同居 | 既存PG資産が大きい、小〜中規模グラフ、PoC |
| SQL Server Graph / Oracle Property Graph | ベンダ組込みのグラフテーブル・クエリ | 既存ライセンス・運用体制の活用 |
| SQL:2023 **SQL/PGQ** | 標準SQLにプロパティグラフクエリが入った | 今後の各DBの実装動向に注目 |

注意点:
- 多くは内部がリレーショナル実装 = **index-free adjacencyではない** → 深い探索の性能はネイティブGraph DBと別物。**期待値を実測で確認**してから採用
- エコシステム(可視化、グラフアルゴリズム、GraphRAGライブラリ連携)は Neo4j 等が先行

<!--
- 「二択ではない」ことを示す現実的なスライド。特にAGEは「Postgresを捨てられないがCypherは書きたい」現場への実用解
- 採用判断の軸を明確に:①グラフのサイズと深さ(小さければAGEで十分)②運用を増やせるか(新DBの監視・バックアップ・スキルは実コスト)③9章以降のLLM連携ライブラリ対応

## 併用アーキテクチャ:全面移行はしない

```
                 書き込み・トランザクション・集計
   業務アプリ ─────────────→ ┌──────────────┐
                             │    RDBMS     │ ← Source of Truth(正)
                             └──────┬───────┘
                                    │ 同期(バッチ or CDC)
                                    ▼
   GraphRAG・エージェント     ┌──────────────┐
   影響分析・不正検知 ──────→ │   Graph DB   │ ← グラフ的クエリ専用の
   (グラフ的クエリのみ)      └──────────────┘    リードモデル
```

- **グラフ的クエリだけ**を Graph DB に切り出す。業務の正データはRDBに残す
- 移行リスクが最小:失敗しても**捨てて作り直せる**(正はRDBにある)
- CQRS の読み取りモデル特化版と考えると設計に馴染む

<!--
- 実務の答えはほぼこれ、と言い切ってよい。既存業務システムのRDBを剥がす移行は数年仕事だが、リードモデル追加は数週間で始められる
- 「捨てて作り直せる」が心理的・政治的に効く:上司への提案は「移行」ではなく「グラフビューの追加」と言うべき。ロールバック可能な提案は通りやすい
- CQRSを知っている受講者には一言で通じる。知らない人には「検索用に全文検索エンジン(Elasticsearch)を横に置くのと同じ構図」という比喩が通じやすい
- 本コースの後半(7章〜)で作るシステムもこの構成が前提。RDB/文書 → KG構築 → LLM活用、という流れの置き場所を示す
-->

## RDBからの同期パターン:バッチ vs CDC

| | バッチ(定期ETL) | CDC(変更データキャプチャ) |
|---|---|---|
| 仕組み | 夜間等にSQLで抽出→グラフ再構築/差分投入 | WAL/binlogを購読し準リアルタイム反映(Debezium等) |
| 鮮度 | 日次〜時間単位 | 秒〜分 |
| 実装・運用コスト | **低**(SQLとスクリプトで書ける) | 中〜高(基盤・順序保証・スキーマ変更追従) |
| 向く用途 | ナレッジ検索、定期の影響分析 | 不正検知、リアルタイム影響分析 |

設計の指針:
- **まずバッチで始める**。鮮度要件が本当に厳しいと分かってからCDC
- **冪等な全再構築スクリプト**を必ず持つ(グラフが壊れても作り直せる保険)
- 鮮度のSLA(「グラフは最大◯時間遅れ」)を利用者と合意しておく

<!--
- 「同期は地味だが、併用構成の成否はここで決まる」と重要度を上げて話す
- バッチ推しの理由を丁寧に:GraphRAGやナレッジ検索の大半は日次鮮度で困らない。CDCは魅力的だが、順序保証・削除の伝播・スキーマ変更追従など運用の罠が多い
- 冪等な全再構築はKG運用の生命線:8章(継続更新)・11章(運用設計)で再登場する布石。「差分同期が信用できなくなったら全部作り直す」ボタンがあると夜眠れる
- 鮮度SLAの話は受講者に刺さりやすい:「このグラフ、最新?」と聞かれ続ける未来を防ぐのは技術ではなく合意
-->

1. **PostgreSQL**:再帰CTEで「役員(id=1)から深さ 3 / 5 / 7 の配下全員」を実行し、

## 第3章まとめ

- RDBでのグラフ表現(隣接リスト、閉包テーブル…)は**読み書きのトレードオフ**の工夫。
  再帰CTEが正攻法だが、**深さ・分岐・関係の種類**が増えると急速に苦しくなる
- 本質的な違い:JOINは**毎ホップ O(log N) の計算**、index-free adjacencyは**O(1)のポインタ辿り**
  — 深さ3で3桁の実測差
- 判断は**チェックリスト5項目**(3ホップ+/関係が増える/関係に価値/パターンマッチ/LLM探索)
- ただし**RDBMSで十分なケースを見極める**のも同じくらい重要
- 現実解は**併用アーキテクチャ**:グラフ的クエリだけ切り出し、正データはRDB、同期はまずバッチ

**次章**:Neo4j / Cypher 実践 — 今日「読んだ」Cypherを、書けるようになる
